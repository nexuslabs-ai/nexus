# OKLCH Color Token Migration — Plan

> **Status note:** The visual-regression layer described below (Chromatic +
> `@chromatic-com/storybook` + `packages/react/src/storybook/modes.ts`) was
> wired in alongside this migration, then removed in PR #64 as out of scope
> for the CI quality-gates work. References to Chromatic and the `modes.ts`
> path in the original draft have been struck below. APCA contrast (§6) and
> the L-grid eyeball pass remain the validation layers in use today.

Target value space at consumption time: OKLCH, with lightness pinned to a fixed perceptual grid. Hex sRGB stays as the on-disk source format because design tools (Tokens Studio, Figma Variables) hex-normalise on export and cannot round-trip OKLCH. The migration moves into the build pipeline.

## 2. DTCG format compatibility

**Decision: stay with string-form `$value`; deviate from DTCG-2025.10's structured-object form.**

DTCG-2025.10 (Color Module) prefers a structured `$value`: `{ "colorSpace": "oklch", "components": [0.66, 0.18, 250], "alpha": 1, "hex": "#3b82f6" }`. We don't adopt this. The repo writes color values as raw hex strings in JSON and emits CSS `oklch(...)` strings at build time — both pass through `utils.js:formatTokenValue` unchanged. Reasons: (a) Tokens Studio and Figma Variables write hex strings on export, not DTCG structured objects, so the structured form would be lost on the next round-trip anyway; (b) the resolver requires no schema migration; (c) the spec is still a draft, and the repo controls both ends. Document the deviation in `.claude/rules/tokens.md`. Revisit if a downstream consumer ever needs spec-compliant import.

## 3. Conversion strategy

**Decision: hex remains the source of truth; perceptual-L conversion happens at build time, not in the JSON.**

The original plan to re-author `color.json` directly to oklch is rejected on new evidence: Tokens Studio explicitly converts non-hex colors back to hex on export (issue #2459, closed "not planned"), and Figma Variables stores hex natively. Any "edit in Figma → re-export JSON" cycle would silently clobber oklch values. The fix is to move the value-space change one layer down:

- `color.json` keeps hex. Designers edit hue freely in Figma; only hue and chroma flow through.
- A new module `scripts/lib/perceptual-grid.js` defines the converters; the L grid itself lives in `scripts/lib/perceptual-grid.json` (`L = 0.985 / 0.945 / 0.870 / 0.765 / 0.660 / 0.553 / 0.460 / 0.385 / 0.297 / 0.207 / 0.118` for shades `50 / 100 / 200 / 300 / 400 / 500 / 600 / 700 / 800 / 900 / 950`) so designers can re-tune without a script-code PR.
- `utils.js:formatTokenValue` is extended: when `type === 'color'` and the value is a hex string, route it through one of two converters:
  - **Grid-pinned** for primitive palette shades (key matches `^(50|100|200|300|400|500|600|700|800|900|950)$` under a palette group in `color.json`). Hue/chroma come from culori, L overwritten by the grid, chroma clamped via `culori.toGamut('rgb')`. _(Post-merge: the clamp shipped as `clampChroma(target, 'oklch', 'rgb')` and was retargeted to `'p3'` by [#86](https://github.com/nexuslabs-ai/nexus/issues/86); live spec in [`.claude/rules/tokens.md`](../../.claude/rules/tokens.md).)_
  - **Mechanical** for everything else (semantic overlay hex with alpha, white, black, any one-off). Straight hex→oklch via culori, alpha preserved.

Mechanical-for-all is rejected because it preserves Tailwind's lightness drift (shade 500 ranges ~0.55 in slate to ~0.66 in yellow), which is the exact defect motivating the migration. The L-grid discipline lives in code now, not in JSON — designers can change hue without breaking it. That is the design discipline the migration is meant to encode.

## 4. Browser support + fallback

**Decision: drop pre-OKLCH browsers. No fallback emitted.**

OKLCH is Baseline 2023 — Chrome 111 (Mar 2023), Safari 15.4 (Mar 2022), Firefox 113 (May 2023), Edge 111, Samsung Internet 22. The narrower `oklch(from …)` relative-syntax isn't used here, so the broader Baseline floor applies. Nexus's only consumer surfaces in-repo are `apps/playground` (internal theme demo) and `apps/docs` (placeholder per root `CLAUDE.md`). No marketing surface. `@supports`-paired blocks would roughly double the CSS size of every color-bearing rule for a population the consumers don't ship to; build-time hex fallback is worse because it re-introduces the lightness drift on the same browsers the fallback is meant to support. Document the floor in `README.md`. Consumers needing older browsers pin to the last pre-migration tag.

## 6. Testing

**Decision: two layers, one new dep.** (Originally three layers; the visual-regression layer was dropped — see status note at top.)

1. **Contrast — APCA, not WCAG 2.** WCAG 2's sRGB-luminance formula treats hues equally; human perception doesn't (blue under-contributes to perceived luminance). APCA fixes that. APCA isn't officially tied to OKLCH, but for an OKLCH-authored palette it's the right tool; WCAG-2 audits would be silently misleading. Add `apca-w3` as a `packages/core` dev dep (~6 kB, zero transitive deps). Write `scripts/audit-contrast.js` to walk every `base-*-{light,dark}.json` and assert `|Lc| ≥ 60` for `foreground ↔ background`, `muted-foreground ↔ muted`, `primary-foreground ↔ primary-background`, plus the equivalents in every brand file. Wire into `yarn build` and CI.

2. **Perceived-lightness eyeball pass.** Render each of the 22 palettes as an 11-shade vertical strip in both themes. The grid is wrong if any palette's progression looks accelerated or stalled. Mandatory before designer sign-off; no tooling.

## 7. Rollout

**Decision: single-branch, single-merge. No hex/oklch coexistence.**

Ordered steps:

1. Branch from `main`: `tokens/oklch-migration`.
2. Add `culori` and `apca-w3` as `packages/core` dev deps.
3. Write `scripts/lib/perceptual-grid.js` (L grid + two converter functions). Extend `formatTokenValue` in `utils.js` to route `$type: "color"` hex values through the converter. Pattern-match palette shade keys for grid-pinned routing; mechanical for everything else.
4. Run `yarn tokens:tailwind` + `yarn tokens:modular`. Confirm JSON files unchanged on disk; generated CSS now contains `oklch(...)`.
5. Write `scripts/audit-contrast.js`. Run it. Fix any APCA failure by adjusting chroma at the offending palette/shade (in the source hex, since H/C come from there) — never by abandoning the L grid.
6. Eyeball the 22 palette strips. Designer reviews.
7. Update `.claude/rules/tokens.md` (hex-in / oklch-out contract; APCA gate; L enforcement note for designers), `packages/core/CLAUDE.md`, `README.md` browser-support floor.
8. Merge.

**Rollback signal**: if designer rejects the L grid on > 2 palettes, do not revert the architecture — re-tune the grid constants in `scripts/lib/perceptual-grid.json`, regenerate, re-review. The architecture's value is independent of the specific L values chosen.

## 8. Traps

- **L is silently overwritten for palette shades.** A designer picking a hex in Figma sees their chosen L lost on build — only hue and chroma flow through. This is the discipline the migration encodes, but it surprises designers who don't know about it. Call it out in `tokens.md` and in any onboarding doc for the Figma workflow.
- **Gamut clip on vivid shades.** Some current hex values (notably `yellow.400-500`, `lime.400`, `fuchsia.500`) sit outside displayable sRGB at the pinned L. `culori.toGamut('rgb')` clamps chroma while preserving L/H — accept saturation loss, not L drift. The converter should warn (not fail) when post-clamp chroma drops by > 20%; those palettes get a manual hue nudge in the source hex. _(Post-merge: the gamut step was retargeted from sRGB to Display P3 by [#86](https://github.com/nexuslabs-ai/nexus/issues/86); the "warn (not fail)" intent stated here was preserved over a hard throw.)_
- **Don't rename the 50–950 scale.** The token name is the consumer contract; the value space is private.
- **Don't split conversion approaches across palettes.** One L grid for all primitive palette shades. A mechanically-converted "exception palette" would silently re-introduce the drift.
- **Overlay tokens carry alpha.** Hex `#000000cc` (alpha 0xCC = 0.8) must emit `oklch(0 0 0 / 0.8)`. The mechanical converter must extract alpha from 8-digit hex — culori handles this, but the routing logic has to actually call it for non-shade keys.
- **Tailwind v4 is not the risk.** Native palette is oklch since v4.0 (Jan 2025 blog); `var(--nx-color-*)` resolves regardless of value format. Don't pin a Tailwind version on the basis of OKLCH support.
- **DTCG-2025.10 structured-object form is not adopted.** If a future consumer wants to import tokens spec-compliantly, the converter needs a second emit path. Note this in `tokens.md` so the deviation isn't accidentally cargo-culted into a new repo.
