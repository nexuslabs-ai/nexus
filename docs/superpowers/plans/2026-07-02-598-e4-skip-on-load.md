# E4 — Skip Animation on Load Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Workstream E principle #13 (part of issue #598) — stop the Avatar fade from firing spuriously on first page paint. The `AvatarFallback` (initials) and `AvatarImage` currently run an unconditional mount `animate-in`, so every avatar on a page fades in on load.

**Architecture:** The literal `initial={false}` mechanism requires Framer Motion, which this repo does not use — so a full first-render guard is out of scope (a JS-ref follow-up, if ever wanted). The **concrete, defensible CSS change** is: remove the mount fade from `AvatarFallback` (it is the default/initial content — fading it on page load is exactly the spurious animation #13 targets), and keep `AvatarImage`'s fade but tie it to the image's loaded state so it reads as "image loaded in," not "component mounted." All portal overlays are already correctly gated by `data-[state=open]` (mount-on-open), so they need no change.

**Tech Stack:** React, Tailwind v4 (`nx:` prefix), Radix Avatar, Storybook 10 (`storybook/test`).

## Global Constraints

- Keep `nx:motion-reduce:*` behaviour intact.
- `nx:` prefix; semantic tokens only. **Pre-production.** **PR:** `feat(motion): …`, base `main`, body references `#598` (sub-PR 4 of 4, `Closes #598`).

### Task 1: Remove the spurious fallback fade; tie the image fade to load

**Files:** `avatar.tsx:166,200`, `Avatar.stories.tsx`.

- [ ] **Step 1: Write the failing test** — add to `Avatar.stories.tsx`:

```tsx
export const NoFallbackLoadFlash: Story = {
  render: () => (
    <Avatar>
      <AvatarFallback>JD</AvatarFallback>
    </Avatar>
  ),
  play: async ({ canvasElement }) => {
    const fallback = canvasElement.querySelector(
      '[data-slot="avatar-fallback"]'
    );
    // Fallback (initials) must not fade in on page load.
    await expect(fallback).not.toHaveClass('nx:animate-in');
    await expect(fallback).not.toHaveClass('nx:fade-in-0');
  },
};
```

- [ ] **Step 2: Run — FAIL.** `pnpm test:storybook avatar`

- [ ] **Step 3: Implement.**

`AvatarFallback` (line 200) — **remove** the mount-fade line entirely, leaving only the layout line:

```tsx
      className={cn(
        'nx:flex nx:size-full nx:items-center nx:justify-center nx:rounded-[inherit] nx:bg-muted nx:text-foreground nx:font-medium nx:leading-none',
        className
      )}
```

`AvatarImage` (line 166) — keep a fade but drive it from the loaded state instead of a mount keyframe, so it plays only when a real image resolves (Radix renders `AvatarPrimitive.Image` once loaded and exposes `data-state`):

```tsx
      className={cn(
        'nx:aspect-square nx:size-full nx:rounded-[inherit] nx:object-cover',
        'nx:opacity-0 nx:transition-opacity nx:duration-fast nx:ease-enter nx:data-[state=loaded]:opacity-100 nx:motion-reduce:transition-none',
        className
      )}
```

(If, in review, Radix does not expose `data-[state=loaded]` on the rendered `<img>`, revert `AvatarImage` to its original `animate-in` line — it only fires on genuine load anyway — and keep just the fallback removal, which is the core #13 fix.)
