**Date:** 2026-07-07
**Status:** Design — scope decisions resolved 2026-07-07 (revised after a 4-agent review council: architect · sde2 · tester · Explore, all findings verified to `file:line`; D1/D2 scope calls set by the user)
**Issue:** #623 — "Add ChoiceCard rich checkbox/radio option cards" (OPEN)
**Branch:** `codex/issue-623-choice-card`

**Goal:** Implement `ChoiceCard` — the rich title/description selectable card for checkbox/radio choices — as a native `Label` shell wired by `htmlFor`, with the nested `Checkbox`/`RadioGroupItem` remaining the only focusable, interactive, source-of-truth control. Checkbox/radio primitives and `ChoiceRow` are unchanged.

**Architecture:** ChoiceCard borrows its **composition + a11y contract** from `ChoiceRow` (native `<label>`, `htmlFor`/`id` wiring, consumer-supplied aria, group-based disabled dimming) but its **visual card box and selected-state highlight** from `FieldLabel`-as-card (`field.tsx:174-175`) — which it must then _improve on_ by scoping the `:has()` selectors to first-party direct children. The card tint is **decorative/progressive** (`:has()` is below the Firefox floor); the atom indicator is the real state signal.

**Tech Stack:** React, Tailwind v4 (`nx:` prefix, `:has()`/`not-has`/`group-has` variants), Radix `Label`, CVA-free (no enum variants), Storybook 10 (`storybook/test`).

## Decisions (resolved 2026-07-07)

Both council blockers below were resolved by the user (via Codex), who adopted the testing / selector / a11y / browser-floor corrections and set the two scope calls. Recorded here with rationale so the intent survives.

### D1 — ChoiceCard is the _preferred_ rich card; `FieldLabel + Field` stays as a lower-level primitive. **RESOLVED**

`FieldLabel` already ships a component-level selectable-card highlight — `nx:has-data-[state=checked]:border-border-primary nx:has-data-[state=checked]:bg-primary-subtle` (`field.tsx:174-175`) — with a `SelectableCard*` suite (`Checkbox.stories.tsx:348-449`) and `Checkbox` JSDoc pointing at it (`checkbox.tsx:28-33`). The council flagged this as "two blessed APIs for one job" and recommended deleting FieldLabel-as-card. **That fork is rejected as too broad for #623**, which explicitly keeps the `FieldLabel + Field` examples in `Field.stories.tsx` as lower-level composition.

Resolution: the two are positioned at **different levels**, not treated as duplicates. ChoiceCard is the preferred rich checkbox/radio option-card API; `FieldLabel + Field` remains the lower-level field-composition primitive. The disambiguation lives in **docs/JSDoc**, which stop presenting `FieldLabel + Field` as the recommended rich-card path.

- **Do NOT** delete `FieldLabel`'s card styling (`field.tsx:174-175`) or drop ChoiceCard.
- **`Checkbox.stories.tsx`:** remove the rich `SelectableCard*` examples (`:348-449`) so it stays atom-focused, and delete their now-orphaned helpers (`BOTTOM_BORDER_ROW` / `BORDERLESS_ROW` / `FrameSection`, `:40-60`) in the same change. Their `FieldLabel + Field` pattern is already covered by `Field.stories.tsx`.
- **`Field.stories.tsx`:** keep `CheckboxCard` (`:236`) and the `FieldLabel + Field` examples, reframed in prose as lower-level composition (not the preferred rich-card API).
- **`ChoiceCard.stories.tsx`:** author _fresh_ stories (not a copy of the FieldLabel+Field ones).
- **Rejected:** deleting FieldLabel-as-card (too broad); `variant="card"` on ChoiceRow (would fork ChoiceRow's body; a sibling is cleaner).
- **Pre-existing, out of scope:** `FieldLabel:175`'s selector is _unscoped_ (`has-data-[state=checked]` matches any nested checked descendant — a nested Switch/Accordion would tint the card). Not introduced by this PR and #623 keeps FieldLabel as-is; note it as a latent follow-up, do not fix here.

### D2 — Shell-state tests use computed-style comparison between cards. **RESOLVED**

`toHaveClass('…has-[…]:border-…')` is **vacuous** — the utility string sits in `class` whether or not `:has()` matches at runtime, so it cannot distinguish selected from unselected (tester, verified). Resolution: assert the shell state via **computed-style comparison between selected / invalid / unselected cards** — not `toHaveClass`, not a hard-coded oklch literal.

- Assert **inequality** between cards: a selected card's resolved `border-color` differs from an unselected one, and an invalid card's differs from a selected one. Robust because `border-primary` / `border-default` / `border-error` are genuinely distinct tokens — this sidesteps the "computed-color asserts fragile" precedent (two semantic tokens collapsing to one value per theme); the one caution to keep if a future palette makes two of them coincide.
- Keep the **atom `data-state`** assertions as the source-of-truth signal (as `ChoiceRow.stories.tsx:106-111,157-159` already do); the computed-style checks prove the _decorative_ shell actually reacts.
- **Relies on the real-browser story runtime** (`test:storybook` → browser-mode Vitest), where `:has()` genuinely evaluates — this would not work under jsdom.
- **Rejected:** reflecting `data-selected` on the root via JS — it forces ChoiceCard to track the control's state (context or DOM observation), contradicting the pure-CSS, control-is-source-of-truth design.

## Global Constraints (from #623)

- **Atoms frozen:** no behavior change to `Checkbox`, `RadioGroup`, `RadioGroupItem`, or `ChoiceRow`. **`ChoiceRow` stays unchanged — including its props type (do not add the `asChild` Omit to it).** Field/FieldLabel are touched **for docs/story framing only** — no behavior or styling changes; see D1.
- **Static content only in V1:** no embedded links, buttons, menus, or actions inside the card. **Verified correct, not a limitation to fix** — inside a native `<label>`, label activation would hijack a nested link/button click (`FieldDescription` supports `<a>` at `field.tsx:274`; that content model does not transfer safely into the label shell).
- **`asChild` omitted** — see Task 1.
- `nx:` prefix before every modifier; semantic tokens only; Tailwind utilities only.
- **`:has()` shell styling is decorative/progressive** — the atom indicator must communicate selected/focus/disabled/invalid with no JS fallback (§ Browser-Floor Decision).
- **Tests are stories.** Pre-production: change in place. PR: `feat(react): add ChoiceCard`, base `main`, body `Closes #623`, Summary + Test Plan + polish.md evidence.

## Task 1 — ChoiceCard component (composition + a11y contract)

**Files:** `choice-card/choice-card.tsx`, `choice-card/index.ts`, `src/index.ts`.

- [ ] **Props type.** `type ChoiceCardProps = Omit<React.ComponentProps<typeof Label>, 'asChild'>`.
  - `asChild` **is** a real prop on `Label` (Radix `Primitive` chain; verified), so the Omit is meaningful — **and #623 mandates it**: dropping `asChild` prevents a consumer swapping the native `<label>`, which would break the `htmlFor`+nesting a11y contract.
  - This **diverges** from `ChoiceRowProps` (plain `extends React.ComponentProps<typeof Label>`, `choice-row.tsx:11`) — **the divergence is intentional and stays.** Do **not** reconcile by adding the Omit to `ChoiceRowProps`: #623 requires `ChoiceRow` unchanged, and even a type-level prop removal is a public-API change. `ChoiceCard` omits `asChild` because #623 requires native-label semantics; `ChoiceRow` is untouched in this PR. (Here the ripple-effect instinct yields to the issue's explicit freeze on the sibling.)
- [ ] **Shell** (`ChoiceCard`): render `<Label data-slot="choice-card" className={cn('nx:group/choice-card …', className)} {...props} />`. Base classes (FieldLabel:174 precedent, **not** ChoiceRow):
  - box: `nx:box-border nx:flex nx:w-full nx:min-w-0 nx:cursor-pointer nx:items-start nx:gap-3 nx:rounded-md nx:border-default nx:border-border-default nx:p-4 nx:transition-colors`
  - (natural height — no `min-h-*`; `items-start` so a tall description top-aligns the marker.)
- [ ] **Subcomponents.** `ChoiceCardContent` (`div`, `data-slot="choice-card-content"`, `nx:flex nx:min-w-0 nx:flex-1 nx:flex-col nx:gap-0.5`), `ChoiceCardTitle` (`div`, `data-slot="choice-card-title"`, `nx:min-w-0 nx:typography-label-default nx:text-foreground`), `ChoiceCardDescription` (`p`, `data-slot="choice-card-description"`, `nx:typography-body-default nx:text-muted-foreground`).
  - **Keep these separate from ChoiceRow's** (do not share/abstract yet — user decision): they look similar, but ChoiceCard owns different layout/state responsibilities, and abstracting now would save little while blurring the two sibling APIs. Duplicate the four subcomponents.
- [ ] **Disabled dimming** (was missing): add `nx:group-has-[:disabled]/choice-card:text-disabled-foreground` to Title and Description, mirroring `choice-row.tsx:76,98`.
- [ ] **Barrel + export**: create `index.ts`; insert the alpha-ordered re-export at `src/index.ts:25`.
- [ ] **a11y invariant (state it, don't just assume it).** JSDoc must teach the required consumer wiring — `id` on the control, `aria-labelledby`→**title id**, `aria-describedby`→**description id** — and _why_: the native `<label>` wraps both title and description, so without `aria-labelledby` the accessible name becomes the concatenated `title + description`. Model the JSDoc example on `ChoiceRow`'s (`choice-row.tsx:21-27`) but show the description wiring (which ChoiceCard makes first-class).

## Task 2 — State model (`:has()`, scoped + guarded)

**Files:** `choice-card/choice-card.tsx`.

> Verify every generated class actually emits (grep the built CSS for the utility, not just the `@theme` var — Tailwind v4 tree-shakes unreferenced utilities). Exact class strings are worked out here during TDD; the **mechanism** is fixed:

- [ ] **Hover/active** — gated off disabled: `nx:not-has-[:disabled]:hover:bg-background-hover nx:not-has-[:disabled]:active:bg-background-active` (ChoiceRow idiom, `choice-row.tsx:35`).
- [ ] **Disabled** — `nx:has-[:disabled]:cursor-not-allowed` (+ any shell dim). Native `:disabled`, not a slot.
- [ ] **Selected / indeterminate** — **direct-child, guarded** so it loses to invalid and disabled:
  - checkbox checked: `nx:has-[>[data-slot=checkbox][data-state=checked]]:not-has-[:disabled]:not-has-[>[aria-invalid]]:border-border-primary` (+ `bg-primary-subtle` if desired).
  - checkbox indeterminate: same shape with `[data-state=indeterminate]`.
  - radio checked: same shape with `[data-slot=radio-group-item][data-state=checked]` (radio has **no** indeterminate — `radio-group.tsx` styles only `checked`).
- [ ] **Invalid** — direct-child, guarded off disabled, **wins over selected**: `nx:has-[>[aria-invalid]]:not-has-[:disabled]:border-border-error`. This realizes #623's "invalid selected keeps the invalid border" that the naive cascade breaks.
- [ ] **Unrelated-nested guard is satisfied structurally** by the `>` direct-child combinator above — reconcile with the V1 "no nested interactive content" assumption (the guard is belt-and-suspenders; the test uses a decoy — Task 4).

## Browser-Floor Decision (polish.md — mandatory)

- **Feature:** CSS `:has()` for the card shell's selected/invalid/disabled/hover state.
- **Floor check:** `:has()` shipped in **Firefox 121** (2023-12-19); the Nexus floor is **Firefox 113** — so on FF 113–120 the shell state does **not** apply. (Chrome 111+, Edge 111+, Safari 15.4+ all support it.)
- **Decision:** ship it as **progressive enhancement**. The real, non-negotiable state signal is the checkbox/radio indicator itself (`checkbox.tsx:86-109`, `radio-group.tsx:84-99`) — fully visible on FF 113–120. The `:has()` card tint is decorative.
- **Precedent:** `choice-row.tsx:35-36` and `field.tsx:175` already use `:has()` below the same floor.
- **Fallback / progressive enhancement:** none needed beyond the always-present atom marker; no JS fallback.
- **Modern Web Guidance:** local-first (no external refresh); the `:has()` progressive-styling guidance is captured here. Sources: MDN Firefox 121 release notes; caniuse `css-has`.
- **Gate:** keep `pnpm audit:browser-support` in the validation run.
