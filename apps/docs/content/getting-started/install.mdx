# Install

Nexus has two delivery models. The **runtime engine** and **lint rules** publish
to npm; the **components** and **Tailwind theme** are copy/own — you vendor their
source into your app and own it from there.

| Package                        | How you get it                                                                                       |
| ------------------------------ | ---------------------------------------------------------------------------------------------------- |
| `@nexus_ds/core`               | `npm install` — framework-agnostic appearance / theme engine                                         |
| `@nexus_ds/eslint-plugin`      | `npm install -D` — lint guardrails for the conventions                                               |
| `@nexus_ds/react` (components) | copy/own — vendored via the export tool ([#541](https://github.com/nexuslabs-ai/nexus/issues/541))   |
| `@nexus_ds/tailwind` (theme)   | copy/own — regenerated with your own token choices                                                   |

> **Copy/own is still landing.** The standalone export tool that copies the
> components and token CSS into a consumer app is tracked in
> [#541](https://github.com/nexuslabs-ai/nexus/issues/541). Until it ships, the
> components are consumed inside this monorepo via the workspace. The published
> `@nexus_ds/core` and `@nexus_ds/eslint-plugin` packages install from npm today.

## Install the published packages

The runtime engine — required whenever you use the Appearance system
(`@nexus_ds/react/appearance`):

```bash
npm install @nexus_ds/core
```

`@nexus_ds/core` owns the appearance model, theme derivation, and the first-paint
bootstrap. Add the lint rules as a dev dependency:

```bash
npm install -D @nexus_ds/eslint-plugin
```

## Vendor the components and theme

`@nexus_ds/react` (the components) and `@nexus_ds/tailwind` (the `nx:`-prefixed
utility theme) are copy/own — you take their source into your app rather than
installing them from npm. The export tool ([#541](https://github.com/nexuslabs-ai/nexus/issues/541))
automates this. The copied React entrypoint needs `react-hook-form` and `sonner`
as peers, which back the Form and Toaster components:

```bash
npm install react-hook-form sonner
```

The sections below show how the vendored packages are wired and consumed under
their `@nexus_ds/*` module names.

## Wire the styles

Import the theme and the prebuilt component styles once, at your app's CSS
entry point:

```css
@import '@nexus_ds/tailwind';
@import '@nexus_ds/react/styles.css';
```

The second import matters: Tailwind doesn't scan `node_modules`, so the
utilities used _inside_ the compiled components ship as prebuilt CSS.

## Declare native color scheme

Add this to your app shell so the browser can theme native controls and the
initial page canvas before JavaScript loads:

```html
<meta name="color-scheme" content="light dark" />
```

`@nexus_ds/tailwind` also emits the matching CSS policy: `:root` advertises
light/dark support, `.dark` pins native UI to dark, and light mode stays pinned
when `.dark` is absent.

## Render a component

Every utility is prefixed with `nx:` and every color is a semantic token, so
components theme themselves. The buttons below are live `@nexus_ds/react`
components rendered straight from this Markdown page:

<div className="nx:flex nx:flex-wrap nx:gap-3 nx:my-6">
  <Button>Primary</Button>
  <Button variant="secondary">Secondary</Button>
  <Button variant="outline">Outline</Button>
  <Button variant="ghost">Ghost</Button>
</div>

Swap the base, brand, or density in the theme picker (bottom-right) and they
re-theme live — no rebuild.
