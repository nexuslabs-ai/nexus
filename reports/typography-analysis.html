4 · Real gaps, restated

Once the report's wrong claims are stripped

Seven concrete findings against the actual code. Sorted by severity.

Gap Kind
01
lyra and mira are vega in costume
Two of the five mode files are byte-copies of vega, with no defined identity. --typography=lyra and --typography=mira are no-op CLI flags. nova / vega / maia themselves are fine — distinct ±1px scaffold across all 13 steps.
 Strategic
02
Letter-spacing pattern is half-applied
Display + heading-xlarge use tight. Heading-large through heading-xsmall use normal. Either commit to proportional spacing throughout the heading tiers, or revert display tier to normal. Don't leave it inconsistent.
 Partial
03
No font-optical-sizing: auto anywhere
All 17 utilities. Inter v4 supports the opsz axis on Google Fonts. Single CSS property; zero bundle cost; activates sharper letterforms at small sizes and more open ones at display sizes.
 Missing
04
No text-wrap: pretty on body utilities
Prevents typographic orphans on body-large / body-default. Modern-browser only but graceful. Zero bundle cost.
 Missing
05
Float-precision artifacts in letter-spacing JSON
Values like -0.800000011920929 and 0.4000000059604645 suggest a lossy round-trip through 32-bit floats. Should be clean (-0.8, 0.4).
 Cosmetic
06
No variable-font axis documentation
Inter is requested with all 9 weights (which triggers a variable-font fetch from Google Fonts), but nx-font-source doesn't document wght or opsz support. This one the report got right.
 Missing
07
No mode → archetype mapping documented
nova / vega / maia ARE real and map cleanly to tool / product / editorial — but the docs don't say so. A consumer who reads the JSON cannot derive the intent. Quick write-up; pure documentation work.
 Missing

5 · Recommended sequence

One strategic call, then polish

The only mode-shaped decision is what to do with lyra and mira. Everything else is bounded polish on already-shipped utilities.

Step 1
Decide what lyra and mira become
(a) Give them a real differentiator. Different typeface (Geist Sans instead of Inter), different weight palette, different letter-spacing curve, or a different scale ratio (e.g., minor third vs perfect fourth between steps). Something that makes them not just "vega in costume."

(b) Delete them. Free under .claude/rules/project-stage.md — pre-prod, no backcompat. Ship nova / vega / maia as the canonical three-archetype set with documentation. Reintroduce when there's a real brand decision behind them.

Lean (b). Three real modes beat five fake ones; (a) is a multi-day exercise that needs design input. (b) is a single commit and unblocks Step 3 documentation.
Effort (b) ≤ 30 min
Effort (a) Multi-day
Step 2
Polish pass on the already-shipped utilities
Add font-optical-sizing: auto to all 17 utilities.
Add text-wrap: pretty to body-* utilities.
Resolve letter-spacing inconsistency: commit to proportional or revert the display tier to normal.
Clean float-precision letter-spacing JSON (-0.800000011920929 → -0.8).
Effort ≤ 1 hour
Risk Very low
Step 3
Documentation
Mode → archetype guide: nova = dense tool (Figma/Linear-style), vega = standard product, maia = editorial/document (Notion-style).
Variable-font axis docs (wght, opsz) in nx-font-source.
Storybook specimen story showing all 17 utilities at vega, with mode-switch side-by-side comparison across nova / vega / maia.
Effort 1 session
Blocked by Step 1

What the report says to do that doesn't survive
Rec #1 as written — "ship heading utilities" — they're already shipped. Only the polish bits (Step 2) survive.

Role-based MD3/Spectrum aliases — existing utility names are already role-based (label-default, body-default, heading-medium). Aliases are duplication.

Headline judgment

The report was right about the modes: nova / vega / maia are three real density archetypes with a clean ±1px scaffold across every size step. Keep them. The honest scope of the "5 modes" critique is much narrower than I first wrote — it's about lyra and mira specifically, both byte-copies of vega with no defined identity.

The report was wrong about heading utilities. They're already shipped, all 7 tiers from display-large down to heading-xsmall. Rec #1 as written (add heading utilities) is solving a problem that doesn't exist; only the polish bits — font-optical-sizing, text-wrap: pretty, letter-spacing consistency — are net-new.

The single decision worth running first is Step 1: either differentiate lyra/mira or delete them. Delete is a 30-minute commit; differentiate is a multi-day design exercise. Either is defensible — but leaving them as silent vega-copies is the worst of both worlds.
