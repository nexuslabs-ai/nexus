The big idea, in one breath

An app's UI is a stack of "surfaces" — page, card, popover. PR #133 fixed how those surfaces communicate "raised" in both light and dark modes, gave the navigation chrome its own paint set, and codified the rules so future contributors don't have to guess.

Change #1 — Light mode adopts Model 2 (white canvas, shadow lifts)

shadcn's default makes cards slightly grayer than the page. Linear / Notion / Vercel make them all white and let shadow do the lifting. PR #133 switches to the second pattern.

Before — shadcn pattern
Card looks slightly gray on white background: slate.50 on white canvas
Card edges read flat; no shadow
After — Model 2
Card is white, shadow lifts it container: white  ·  canvas: slate.50
Card floats above muted canvas

Change #2 — Dark mode container fix (the real bug)

The pre-PR dark mode had container = #000000 on top of a #0f172a page. Pure black is darker than dark gray, so cards visually sank into the page instead of floating above it. Cards now sit at slate.800 — one step lighter than the canvas — and read as raised.

Before — sinking
Card below page container: black · background: slate.900
Card sinks into the canvas
After — rising
Card above page container: slate.800 · background: slate.900
Card sits above the canvas

Change #3 — The 5-level surface stack, codified

Every surface in the system now picks one of five levels, defined by luminance step only (never by hue). Light values move toward {white}; dark values move away from {slate.900}.

Token Light Dark Role
popover {white} {slate.700} Floating overlays
container {white} {slate.800} Cards, dialog bodies
background {white} {slate.900} Page canvas
muted {slate.50} {slate.800} Recessed regions
nav-background {slate.100} {slate.950} Fixed chrome (sidebar)

Change #4 — Nav-* gets its own paint set

Six tokens — nav-background, nav-foreground, nav-muted-foreground, nav-item-hover, nav-item-active, nav-border — so the chrome can be styled (or brand-tinted) independently. The namespace deliberately stops at chrome: CTAs inside nav reuse base primary-*; focus rings reuse focus-*.

Design system

Primary buttons placed inside nav still use the base primary-* tokens — nav doesn't ship its own nav-primary-*.

Change #5 — Rule 6: press cue moves off the fill

In dark mode, container-active and popover-active can't step to a distinct shade without colliding with neighbouring surfaces. So when you press a card, the token still resolves (as a stable hook for component code) but the visual affordance comes from nx:active:shadow-inner or nx:active:border-border-active — applied at the :active / [data-state="active"] selector.

Click & hold me

Known overlaps (6 collisions, all intentional)

Some tokens land on the same shade — these fall out of the luminance grammar, not bugs. surfaces.md § Known overlaps now enumerates each one.

dark · container = muted = background-hover = {slate.800}
Context resolves the ambiguity — a row that highlights on hover does not become a card.
dark · container-active = container = {slate.800}
Press cue moves off the fill per Rule 6.
dark · popover-active = popover = {slate.700}
Same shape as container-active; popover surfaces are rarely pressed directly.
light · popover-hover = popover-active = {slate.50}
Near-white compression — no half-step between slate.50 and slate.100.
dark · muted-foreground-subtle = muted-foreground = {slate.300}
No palette step exists between slate.300 (passes APCA) and slate.400 (too dim, Lc -39).
dark · nav-border = nav-item-active = {slate.800}
The selected item reads as "pressed into" the chrome, not raised above it.
