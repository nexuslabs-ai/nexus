01 — Alpha variants · expanded

Solid colors paste. Alpha colors blend.

Every Nexus surface, border, and overlay today is a solid color. When a component sits on a non-white background — a colored sidebar, a tinted card, a photo, a scrolling content area — it pastes onto the surface. Alpha-variant tokens are transparent versions of the same colors that pick up the underlying surface tint, so the component reads as part of the surface, not stuck onto it. This pattern is universal in modern design systems; Nexus is one of the only Tier-A-aspiring systems without it.

How it is today vs after

No alpha tokens exist. To get a translucent surface, developers hand-write rgba(0,0,0,0.18) in component CSS. The value is tuned for one background and looks wrong on every other. Theming is impossible because hex values are baked into source code.

Phase 1 adds every base palette its own alpha scale — gray.a50 through gray.a950, chromatic palettes similarly. The semantic layer gains *-alpha variants for borders, muted surfaces, and overlays. The OKLCH pipeline emits these with correct perceptual blending, not RGB approximation.

Every product Nexus aspires to be like has shipped alpha tokens for a decade. The pattern is the table-stakes precursor to "feels like a real design system." Phase 1 lands these mechanically for all 5 base palettes at once — there's no scenario where adding alpha-variant tokens is the wrong call.

02 — Surface contract · tracked in #80

The surface contract moved into a real engineering issue.

The 4-surface story has progressed beyond preview. Issue #80 specifies the exact token changes per file (Model 2 — pure white canvas, shadow-driven elevation in light mode, dark container moved off black) and the new nav-* namespace with 6 tokens instead of 2.

#80

feat(tokens): adopt Model 2 surface contract + nav-* tokens + dark container fix. Resolves three coupled token decisions: (1) Model 2 canvas, (2) Linear-style "bold" nav contrast, (3) dark container fix from black → slate.800. Visual decisioning lives at reports/surface-options.html. Cleanup: removes redundant muted-light. Full token matrix in the issue body.

Key things to know about what #80 actually does

The biggest change isn't structural — it's that dark mode currently inverts elevation. container is set to pure black, which is darker than the dark-mode canvas (slate.900). Cards visually recess instead of rising. Issue #80 fixes this by moving dark container to slate.800 and dark popover to slate.700 so the elevation stack reads correctly in both themes.

On the nav side, #80 chose Option A · Bold (Linear-style: nav one step darker than canvas in both modes) over the "muted" option (nav same as canvas) and "subtle" option (slight separation). Visual rationale is in surface-options.html — open it side-by-side with the issue for the full picture.

Status: open issue, implementation order specified in 12 steps, APCA must pass on regeneration. Track in #80.

1. @nexus/colors — standalone color package

Today Nexus's colors live inside @nexus/tailwind. That package ships variables.css with every CSS custom property baked in — colors, shadows, typography, spacing, radius, border-widths — and carries a Tailwind v4 peer dependency. If you're not using Tailwind v4, you can't cleanly use Nexus colors.

Who's blocked today

React Native developers
Want Nexus blue in a StyleSheet. Today: hand-copy hex values from variables.css. No theming. Web and native drift apart silently the moment a token changes.
Vanilla HTML / static sites
Marketing pages, docs sites, landing pages. Today: install @nexus/tailwind (~18 KB) just for colors, ignore everything else, carry a Tailwind v4 peer dep on a static page.
Svelte / Vue / SolidJS apps
Using their own style systems. Want Nexus colors. Today: same blocker as static sites — bring the whole Tailwind theme or copy hex by hand.
Figma plugin developers
Building plugins that read Nexus token values for design tooling. Today: parse @nexus/tailwind's CSS file or hardcode hex strings — no machine-readable manifest exists.

What changes

A new workspace package — @nexus/colors — that ships just the color tokens. No shadows, no typography, no Tailwind peer dependency. Available as CSS imports per base+theme, or as a JSON manifest for JS consumers.

The pattern this mirrors

Radix's most-cloned move was unbundling @radix-ui/colors from @radix-ui/themes. The colors package has wider adoption than the theme package because color is more universal than component frameworks. Vercel/Geist publishes the same way at vercel.com/geist/colors. Nexus joining this pattern unlocks every consumer outside the React + Tailwind happy path.

Color math credibility doc — Phase 1 unique deliverable

This is the Stripe move. Stripe's 2019 "Designing accessible color systems" post is still cited 7 years later because it published the methodology, not just the result. Nexus has done equivalent engineering (OKLCH at runtime, perceptual lightness pinned per shade, APCA per-tier gating) but has published nothing.

Now Engineering work is invisible
Designers, prospects, and new hires see hex values in JSON files. The OKLCH conversion, the perceptual-grid lightness pinning, the APCA gating — all internal. There's no doc to cite, no proof of taste behind the math, no credibility surface. The engineering effort is real; the trust signal is zero.
After One page, three decisions explained

OKLCH + APCA + perceptual lightness pinning

Three engineering decisions, one consequence: every Nexus shade at the same step has the same perceptual lightness, regardless of palette. Stone-500 is as light as slate-500 is as light as neutral-500.

Continues with: hex-on-disk rationale, the perceptual-grid.json file, APCA threshold tiers, browser support floor, OKLCH vs HSL vs LAB tradeoffs.

Cost: ~2 hours AI draft + ~2 hours human voice review. Value: the engineering becomes citable. Designers can defend Nexus's choices to brand stakeholders. Prospects see the work behind the system, not just the output.
