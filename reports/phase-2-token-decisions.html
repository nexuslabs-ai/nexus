1 — Focus Ring Technique

Nexus currently uses a single-shadow neutral-grey ring (option A below). The Phase 1 report's Rec #1 proposes adding a background-colour spacer (option C). Two problems with that proposal were identified in the cross-check: the spacer doesn't work on coloured surfaces (the case it was supposed to fix), and box-shadow-based focus rings are invisible in Windows High Contrast Mode. Five techniques are compared side-by-side below. Recommendation: Option B (outline + offset) — the same pattern Linear, Stripe, Figma, and Spectrum ship. One CSS declaration pair delivers surface-invariance, a visible floating gap in normal mode, and native WHCM survival, while making issue #3 (Tabs shadow stacking) dissolve as a side effect.

The problem

A focus ring has to satisfy three properties at once:

  1. Visible on every surface — page background, card, muted, primary (dark), error (red), etc. A neutral grey APCA-gated to Lc ≥ 45 covers this in Nexus today.
  2. Visually separated from the element — a ring that sits flush with the element edge is harder to read than one with a 2px gap. This is the "ring offset" property.
  3. Survives Windows High Contrast Mode — WHCM (a.k.a. forced-colors: active) strips box-shadow entirely. A pure-shadow ring becomes invisible for users who depend on the high-contrast theme.

Nexus's current implementation handles (1) — Issue #44 verified APCA Lc ≥ 45 on all semantic surfaces. It does not yet handle (2) or (3).

The five candidate techniques

Each labelled A–E. A is the current state; B–E are candidates.

A · Current state baseline

Single box-shadow at 2px spread, flush with element edge.

outline: none;
box-shadow: 0 0 0 2px
  var(--focus-default);
Pros
  • Already shipped
  • APCA-verified on all surfaces
Cons
  • Invisible in WHCM
  • No visual separation from element edge
  • Stacks with elevation shadows
B · Outline + offset pick

Native CSS outline with a 2px offset. The pre-box-shadow web standard — and what every Tier-A benchmark actually ships.

outline: 2px solid
  var(--focus-default);
outline-offset: 2px;
Pros
  • Visible 2px gap to all users in normal mode (not just WHCM)
  • Survives WHCM natively (outline preserved)
  • Doesn't stack with box-shadow elevation — fixes Tabs §3 automatically
  • One CSS declaration pair; single-token mental model
  • Matches Linear, Stripe, Figma, Spectrum (all use outline)
Cons
  • Drops the unused --nx-focus-geometry-* primitives
  • Refactor across ~8 component files (one-time cost)
  • Outline not composable into asymmetric shapes (theoretical for Nexus)
C · BG spacer (report's Rec #1) skip

Two-layer shadow: inner = page bg colour, outer = ring colour.

outline: none;
box-shadow:
  0 0 0 2px var(--bg),
  0 0 0 4px var(--focus-default);
Pros
  • Minimal token-level change
  • Matches shadcn's ring-offset idiom
Cons
  • Spacer is wrong colour on primary / error / coloured buttons — see demo
  • Still invisible in WHCM
  • Still stacks with elevation shadows
D · Double-ring lightness-inverse candidate

Two rings; outer = inverse lightness of inner. Guarantees one ring has contrast on any background.

box-shadow:
  0 0 0 2px var(--ring),
  0 0 0 4px oklch(from
    var(--ring) calc(1-l) 0 h);
Pros
  • Truly surface-invariant by construction
  • Uses modern OKLCH relative colour syntax
Cons
  • Two visible rings = visually heavier
  • Still invisible in WHCM
  • Requires Baseline 2024 browsers
E · Hybrid fallback

Box-shadow for normal mode + transparent outline as WHCM fallback. The conservative incremental move.

outline: 2px solid transparent;
outline-offset: 2px;
box-shadow: 0 0 0 2px
  var(--focus-default);
Pros
  • Preserves issue #44's box-shadow infrastructure
  • Transparent outline becomes visible system colour in WHCM
  • Smallest delta from current state
Cons
  • Visible gap only in WHCM mode, not normal mode — sighted users still see ring flush with element edge
  • Three CSS declarations vs B's two
  • Still stacks with elevation shadows (Tabs §3 not auto-fixed)
  • Outline / shadow geometry differs between WHCM and normal modes

Recommendation

Pick B · Outline + offset Change the canonical focus pattern in components.md from nx:focus-visible:outline-none nx:focus-visible:shadow-focus-default to: 
nx:focus-visible:outline-2 nx:focus-visible:outline-focus-default
nx:focus-visible:outline-offset-2
One declaration pair, surface-invariant, native WHCM survival, no shadow-stack interaction with elevation. The 2px floating gap is visible to all users in normal mode — not just WHCM users — which delivers what the Phase 1 BG-spacer was trying to achieve, without its coloured-surface failure.
What this drops The --nx-focus-geometry-x/y/blur/spread primitives in variables.css (lines 259–262) and the --shadow-focus-default / --shadow-focus-error composed tokens in nexus.css (lines 205–210). Per .claude/rules/project-stage.md, pre-production code refactors freely — no backward-compatibility shim. Keep --nx-focus-color-default and --nx-focus-color-error (lines 257–258, 431–432); they get promoted to the semantic colour namespace as --color-focus-default and --color-focus-error so Tailwind generates outline-focus-default and outline-focus-error utilities automatically.
Why not E (the conservative move) E preserves issue #44's box-shadow infrastructure but only delivers the visible gap to WHCM users — sighted users still see the ring flush with the element edge. Issue #44's locked rationale was about colour (APCA-gated neutral grey) and avoiding shadow-stacking; the box-shadow-vs-outline axis was tangential. Every Tier-A benchmark that actually documents focus (Linear, Stripe, Figma, Spectrum) uses outline. Reversing #44's tangential mechanism isn't fighting the decision — it's completing the intent.

3 — Tabs Shadow-Stack Issue

TabsTrigger applies nx:data-[state=active]:shadow-sm for elevation and nx:focus-visible:shadow-focus-default for keyboard focus. When a tab is both active and focused, both shadows stack. This violates the components.md rule "Focusable controls do not use shadow elevation" — a pre-existing inconsistency, not one introduced by the Phase 2 work. Adopting Option B in §1 dissolves the correctness side of this issue automatically (outline doesn't compose with box-shadow). The bottom-border recommendation below is now P1 aesthetic polish, not a P0 correctness fix.

The problem visualised

Each tab below is rendered in a specific state. Focus all shows what the user sees when keyboard-focused.

Inactive
Active (shadow-sm)
Inactive + focused (ring only)
Active + focused (sm + ring stacked)
What's stacked in the rightmost case Two box-shadows fire simultaneously: shadow-sm (1px y-offset drop shadow) and shadow-focus-default (2px spread ring). The visual result is a faint drop shadow bleeding out from under the focus ring. It works, but it violates the explicit components.md rule and looks slightly off-spec.

Three resolution options

1 · Strip shadow-sm from active state candidate

Active tab uses border / background change only, no elevation.

Pros
  • Aligns with the components.md rule literally
  • No shadow stacking ever
Cons
  • Loses the "lifted active tab" affordance shadcn ships with
  • Active state becomes less distinct visually
2 · Replace shadow-sm with a bottom-border indicator pick

Active tab gets a 2px bottom border in primary colour; no shadow.

Pros
  • Strong active affordance without shadow stack
  • Matches Linear, Stripe, Notion tab patterns
  • Survives WHCM (border is preserved)
Cons
  • Slightly changes Nexus's tab aesthetic
  • Bottom border interacts with the pill background — needs visual tuning
3 · Accept the stack, update the rule skip

Document Tabs as a deliberate exception to the elevation rule.

Pros
  • Zero code change
Cons
  • Codifies the inconsistency into the rule
  • Future controls will adopt the exception pattern
  • Doesn't resolve the visual issue

Recommendation

Pick option 2 — bottom-border active indicator (now P1, not P0) Strip nx:data-[state=active]:shadow-sm from tabs.tsx:89. Replace with nx:data-[state=active]:border-b-2 nx:data-[state=active]:border-primary-background or equivalent. Makes Tabs follow the "no elevation on focusable controls" rule honestly.
Important: this dropped from P0 to P1 once B (outline) is selected for §1 The shadow-stacking correctness issue dissolves the moment outline-based focus lands — outline doesn't compose with box-shadow, so active-tab elevation and focus indicator no longer interact. The remaining motivation is purely aesthetic (the rule still says "no elevation on focusable controls"). Worth doing, but doesn't need to ship in the same PR as §1 anymore. Can land as a separate small PR after §1 stabilises.

4 — Menu Item :focus Pattern

DropdownMenu and Select items use nx:focus:bg-background-hover — not focus-visible: and not shadow-focus-default. This isn't a bug; it's the correct pattern for Radix's roving-focus menu items. It just isn't documented as such. A new Nexus component author looking at Button and Dropdown side-by-side will see two different focus patterns and not know which to follow.

Why menus use :focus, not :focus-visible

Radix's menu primitives implement roving focus: only one item in the menu has DOM focus at any time, and arrow keys move focus between items. When the user opens a dropdown with a mouse hover, Radix moves DOM focus to the hovered item — by design, because the next keypress (arrow keys) should pick up from where the mouse left off.

This means :focus fires on the item the user is mousing over, not just on keyboard-targeted items. A focus-visible-only ring would be wrong: mouse users would see no indicator while moving through the menu. A background-tint :focus treatment shows the "current item" regardless of input modality — which is exactly the menu UX users expect.

5 — Breakpoints & Responsive Strategy

Nexus inherits Tailwind v4's 5 default breakpoints with zero customisation and zero documented opinion. The cross-check found Dialog already uses nx:sm: in 3 places, which initially read as at odds with the "desktop-primary" stance (a spike later confirmed it's a legitimate viewport-driven exception). Meanwhile 2026 industry consensus has shifted decisively: container queries are now Baseline (Chrome 105+, Safari 16+, Firefox 110+ → ~95% coverage) and have largely replaced viewport media queries for component-internal responsive behaviour. Recommendation: Option D — Token + Documentation + Container Queries default. Promote 5 breakpoint CSS variables to the semantic namespace, document Narrow/Standard/Wide display classes with ≥1024px as the primary target, adopt @container as the canonical pattern for component-internal responsive behaviour, and keep Dialog's three nx:sm: sites as a documented viewport-driven exception (a pre-implementation spike found the naïve @container swap breaks Dialog at 384–640px viewports — see Decision D).

The problem — codebase audit

Verified state (May 2026, this branch):

  • Zero breakpoint tokens — variables.css and nexus.css contain no --breakpoint-* or screens: overrides
  • Zero @container usage anywhere in packages/react/src/
  • Zero dvh / svh / lvh dynamic viewport units
  • Zero clamp() fluid scaling
  • Three nx:sm: sites in dialog.tsx (lines 135, 187, 220) — shadcn-inherited mobile-first defaults
  • No .claude/rules/responsive.md exists; figma.md:329 documents responsive only as "Tailwind responsive prefixes" with no Nexus opinion

The implication: a new contributor (human or LLM) has no Nexus opinion to follow. They default to Tailwind defaults, default to viewport media queries, and default to mobile-first — none of which match how a desktop-tool design system actually performs. The gap is almost entirely documentation, with one piece of seemingly contradictory code — Dialog, which the spike confirmed is a legitimate viewport-driven exception.

Three things worth borrowing from Atlassian's responsive primitive Atlassian's grid system is in beta and being pivoted away from, but their breakpoint primitive (the @atlaskit/primitives responsive layer) is independently strong and stable in shape. Three lessons informed updates below:
  1. rem-based breakpoints, not px — when a user bumps their browser font-size from 16px → 20px, rem-based breakpoints scale proportionally; px-based breakpoints don't. This is the single largest accessibility win in the breakpoint layer, and Polaris (em) + Mantine (em) both ship it. Tailwind defaults to px. Nexus should follow Atlassian's lead — see Decision A update below.
  2. above / only / below on one media object — Atlassian exposes all three media-query intents through a single media object (media.above.md, media.only.sm, media.below.lg). Cleanest API surface for media queries observed. Worth mirroring if Nexus ever ships a JS-side breakpoint helper.
  3. Declarative <Show> / <Hide> primitives — instead of CSS-in-JS or className gymnastics, Atlassian wraps responsive visibility in two small React components. Tiny (each ~30 lines), composable, and remove the most common breakpoint friction in JSX. New sub-decision E below proposes Nexus ship the same pair.

Industry consensus, May 2026

Six load-bearing shifts since the Phase 1 audit was written:

  • Container queries are now Baseline — ~95% browser coverage. The 2026 default for component-internal responsive is @container, not viewport media queries.
  • Viewport breakpoints are demoted to page-skeleton level only — nav, side panels, top-level shell. Components inside should adapt to their container.
  • Dynamic viewport units (svh / lvh / dvh) have replaced raw vh for mobile-edge cases. Default: svh for sticky elements, lvh for modals, dvh only for full-screen hero containers (jitter risk on internal elements).
  • Fluid scaling via clamp() is the modern alternative to breakpoint ladders for typography and spacing — one declaration replaces 3+ media queries.
  • Material 3 expanded window size classes (Oct 2025) — Large (1200–1600dp) and Extra-Large (≥1600dp) added to better target desktop and ultrawide. Total: 5 semantic device classes.
  • Content breakpoints > device breakpoints — 45–75 character reading width (≈600–700px) is the load-bearing constraint, not "what's iPhone width."

For Nexus, this means: viewport breakpoint tokens are still needed (consumers reach for nx:lg: for top-level shell), but the answer to "how do my Nexus components adapt to width" should be @container, not nx:sm:.

The five decisions

Breakpoint strategy isn't one decision — it's five interleaved. Each addressed below with its own recommendation. The end of the section ties them into a single migration plan.

A · Breakpoint token shape CSS variables in @theme

Should Nexus ship breakpoint tokens, or rely on Tailwind defaults?

A1 · Status quo skip

Inherit Tailwind's sm/md/lg/xl/2xl defaults. No Nexus override.

Pros
  • Zero work
  • Tailwind v4 idiomatic
Cons
  • Consumers can't override without Tailwind config
  • No CSS-only consumers (RN, Svelte, vanilla) can adapt
  • Misses 2026 trend toward CSS-variable tokens
A2 · CSS variables in @theme · rem-based pick

Add --breakpoint-sm/md/lg/xl/2xl to nexus.css @theme in rem units. Tailwind v4 auto-discovers them and generates responsive variants. Values scale with user's browser font-size.

Pros
  • Aligns with Tailwind v4's native pattern (auto-discovers --breakpoint-* in @theme)
  • rem-based values scale with user font-size (matches Atlassian, Polaris, Mantine — all em/rem-based)
  • Tailwind's defaults are px; overriding to rem is the accessibility upgrade
  • Build-time consumer override via the consumer's own @theme block — no Tailwind config file needed
  • Values exposed as CSS variables for non-Tailwind contexts (custom @media, React Native, vanilla CSS)
  • One-block change to nexus.css
Cons
  • rem-based breakpoints fire at different px widths for users with non-default browser font-size — usually a feature, occasionally surprising for designers
⚠ What this is NOT

A consumer setting :root { --breakpoint-lg: 70rem } at runtime does not retune Tailwind's generated nx:lg: media queries. @media doesn't accept var() in any current browser — Tailwind bakes the resolved value at build time. The override path is the consumer's @theme block (build-time), not runtime CSS variables. Same constraint applies to container queries (--container-*).

A3 · Numeric (Radix-style) skip

Replace t-shirt names with numeric (1/2/3/4/5).

Pros
  • No device-class implication baked in
Cons
  • Breaks Tailwind familiarity
  • Less actionable for consumers
  • Numeric names lose semantic meaning

B · Semantic display classes + primary target Narrow / Standard / Wide

Should Nexus document semantic intent on top of the breakpoint scale, and state a primary target?

The five Tailwind breakpoints are dimensions without meaning. Material 3 (Compact/Medium/Expanded/Large/XL), Spectrum (S/M/L/XL with iPad target), Linear/Figma/Raycast (all desktop-primary) all attach semantic meaning to their breakpoint range. Nexus should do the same — adapted for its desktop-tool positioning.

Primary target stance — for the Nexus reference brand: Nexus components are designed for Standard (≥1024px). Narrow is graceful degradation; Wide gets extra breathing room when available. Use nx:lg: as the primary responsive prefix in consumer code.

For consumer brands: this is a default, not a mandate. A consumer brand building a mobile-first product can override by setting their own @theme breakpoint values and treating a different tier (e.g., nx:md:) as their primary target. The display-class labels (Narrow/Standard/Wide) are documentation, not hard-coded behaviour — they describe Nexus's reference defaults, which consumer brands can remap. No Nexus utility is named nx:standard: or similar; only the standard Tailwind class names (nx:sm:, nx:md:, etc.) exist as code.

Why Narrow / Standard / Wide and not Material's Compact / Medium / Expanded: MD3's labels assume mobile-first (Compact = phone portrait, 0–600dp). Nexus's reference brand is desktop-tool primary — the same semantic spectrum but rotated 180°. Narrow / Standard / Wide makes the desktop-first default load-bearing in the names without forcing it on consumer brands.

Designer note — zoom and rem interaction Because breakpoints are rem-based (Decision A2), they fire at the user's zoom-adjusted threshold, not pixel-adjusted. A low-vision user at 200% browser zoom on a 1280px viewport sees layouts triggering as if the viewport were ~640px (Narrow). This is the intended accessibility behaviour — content stays at readable proportions when zoomed — but it can surprise designers who test at 1280px expecting Standard. Test at multiple zoom levels and font-size settings, not just default 100% at 16px root.

C · Container queries — when to use, when to skip default for component-internal

Should Nexus components use @container for internal responsive behaviour, or rely on viewport media queries?

2026 default: @container for component-internal width adaptation; viewport nx:sm:/nx:md: only at the page-skeleton level (top-level shell, navigation, side panels). Mantine has it as a first-class concept; Tailwind v4 supports it natively via the @container utility and @sm: / @md: variant prefixes.

PatternUse caseExample
Container query (@container) Component renders differently based on its parent width — works at any viewport A Card that's full-width in a hero, narrow in a sidebar — uses the same JSX
Viewport media query (nx:lg:) Page-shell decisions — navigation collapse, side panel hide, top-level grid Hide the secondary sidebar below nx:lg:
Fluid scaling (clamp()) Continuous adaptation of size — typography, padding font-size: clamp(1rem, 0.9rem + 0.5vw, 1.5rem)
Dynamic viewport units (svh/lvh/dvh) Mobile browser-chrome accommodation Full-screen modal: min-height: 100dvh

Key rule for Nexus contributors and LLM agents: if a component's responsive behaviour depends on the component's own width, use @container. If it depends on the viewport's width (shell-level decisions), use nx:lg:. Most Nexus component-internal adaptation falls into the first bucket.

Carve-out: full-viewport overlays (Dialog, Sheet, FullScreenOverlay) The "component-internal uses @container" rule has one principled exception: full-viewport overlay components. Their responsive trigger is positioning relative to the viewport (full-bleed sheet vs centred card), not the overlay's own container width — so they stay on viewport breakpoints (nx:sm:). The May 2026 spike that established this — including the width table showing why a naïve nx:@sm: migration would flip Dialog's rounded corners on prematurely (at viewports 384–640px) — is in Decision D below. In-rule documentation lives in .claude/rules/responsive.md § Viewport-driven exceptions.

D · Dialog's nx:sm: resolution keep + document as viewport-driven exception

Three sites in dialog.tsx use nx:sm: mobile-first defaults. Keep, strip, or migrate?

Spike finding — May 2026 The naïve nx:sm: → nx:@sm: migration originally proposed in D3 does not work as written. Tailwind v4's container-query namespace (--container-*) is separate from the viewport namespace (--breakpoint-*), and the two scales have different default values. nx:@sm: fires at container ≥ 24rem (384px), NOT at viewport ≥ 40rem (640px). Walking through Dialog's actual widths:
ViewportDialog widthOriginal nx:sm:Naïve nx:@sm:Wanted
320px20rem (full-bleed)no rounded ✓no rounded (20<24) ✓no rounded
480px30rem (full-bleed)no rounded ✓rounded (30≥24) ✗no rounded
640px+32rem (max-w-lg)rounded ✓rounded ✓rounded
At viewports 384–640px the swap flips Dialog's rounded corners on prematurely. The two scales are not interchangeable. Revised pick: D1 (keep + document as exception).
D1 · Keep + document as viewport-driven exception pick (post-spike)

Leave the three nx:sm: sites in dialog.tsx intact. Document Dialog in responsive.md as a deliberate exception to Decision C's @container-first stance.

Pros
  • Honest about Dialog's actual UX trigger: positioning relative to the viewport, not Dialog's own width
  • Preserves the mobile-bottom-sheet behaviour at narrow viewports
  • Zero code change — the existing behaviour is already correct
  • Becomes a teaching moment for future components: "container queries are the default, but viewport-driven UX (Dialog, Sheet, FullScreenOverlay) is the documented exception"
Cons
  • Adds a carve-out to Decision C — slight cognitive load on contributors
  • Mitigated by explicit documentation in responsive.md
D2 · Strip (consumer DIY) skip

Remove the three nx:sm: sites; let consumers add responsive behaviour.

Cons
  • Dialog at <640px viewport becomes ungainly without consumer fix
  • Loses the mobile-bottom-sheet behaviour users expect
D3 · Migrate to @container rejected after spike

Originally proposed: replace nx:sm: (viewport) with nx:@sm: (container).

Why rejected
  • Tailwind's @sm: fires at container ≥ 24rem (384px), not viewport ≥ 40rem (640px) — different scales
  • At viewport 480px, Dialog's container width is ~30rem (full-bleed) — naïve swap triggers rounded corners during the full-bleed phase
  • Dialog's responsive behaviour is genuinely about positioning relative to the viewport, not Dialog's own width — viewport queries are the correct mechanism
  • Workarounds (arbitrary values, custom container sizes) add token surface without addressing the underlying mismatch
Why this is a principled exception, not a workaround §5 Decision C says "component-internal responsive uses @container." Dialog tests that principle. The honest answer: Dialog isn't responding to its own container — it's responding to whether it occupies the full viewport or a centred slot. That trigger is the viewport, not the container. The exception holds for any full-viewport overlay (Dialog, Sheet, FullScreenOverlay). Component-internal layouts inside non-overlay components still default to @container.

E · Ship <Show> / <Hide> primitives (Atlassian-inspired) propose

Two small components that wrap responsive visibility. Removes the most common breakpoint friction in JSX.

Today, hiding a sidebar below lg in Nexus consumer code looks like: 

<aside className="nx:hidden nx:lg:block">Secondary nav</aside>

It works, but the two-class invert is a small footgun (forgetting nx:hidden leaves it visible at all sizes). Atlassian's <Show above="lg"> and <Hide below="lg"> primitives are declarative wrappers around the same CSS — clearer intent, less footgun: 

<Show above="lg">
  <aside>Secondary nav</aside>
</Show>

<Hide below="md">
  <span>Extra context only on tablet+</span>
</Hide>

Nexus extension: ship a third prop for container queries, not just viewport. This is something Atlassian doesn't have — and it's a natural fit for the @container-default stance from Decision C: 

<Show containerAbove="md">
  <span>Visible when the parent container ≥ md, regardless of viewport</span>
</Show>

Implementation is small — each component is ~30 lines of CSS-driven JSX (just toggles display). Children always render (no unmount); only visibility changes. Both components live in packages/react/src/components/primitives/.

API surface (proposed)
PropTypeNotes

below"sm" | "md" | "lg" | "xl" | "2xl"Viewport max-width — show/hide below this breakpoint
containerAbove"sm" | "md" | "lg" | "xl" | "2xl"Nexus extension — container min-width via @container
containerBelow"sm" | "md" | "lg" | "xl" | "2xl"Nexus extension — container max-width via @container

Exactly one of above, below, containerAbove, containerBelow must be specified.

Why this is a Nexus genuine differentiation, not just a copy Atlassian's <Show> / <Hide> only do viewport. Nexus's add containerAbove / containerBelow for container queries. This makes the primitives the FIRST place where a Nexus consumer sees the "container-over-viewport" choice — the same stance Decision C documents in prose, here surfaced in the API. Humans and LLM agents both encounter it at the moment of use.

Step 1 — Add 5 breakpoint CSS variables to nexus.css (rem-based)

/* In @theme block, alongside --shadow-*, --radius-*, --z-* */
--breakpoint-sm:  40rem;   /* 640px @ 16px root  · narrow */
--breakpoint-md:  48rem;   /* 768px @ 16px root  · narrow */
--breakpoint-lg:  64rem;   /* 1024px @ 16px root · Standard floor ★ */
--breakpoint-xl:  80rem;   /* 1280px @ 16px root · Standard reference ★ */
--breakpoint-2xl: 96rem;   /* 1536px @ 16px root · wide */

Tailwind v4 picks these up automatically (it scans @theme for --breakpoint-*). Values map exactly to Tailwind defaults at 16px root font-size, but expressed in rem so they scale with the user's browser zoom and font preferences — same pattern Atlassian, Polaris, and Mantine use.

Anti-slop notes for LLM agents implementing this
  • Tailwind v4 container-query syntax is @container (parent) + @sm: / @md: (children) — note the @ prefix differs from viewport sm: / md:.
  • With the nx: prefix order rule (components.md §145), the correct form is nx:@sm:flex-row, NOT @nx:sm:flex-row. The nx: always comes first.
  • Do NOT add container-type: inline-size by default on every component — only where a child needs to query the container. Carries minor formatting-context cost.
  • Do NOT replace ALL nx:sm: / nx:md: globally — viewport breakpoints are still correct for shell-level page decisions. Only component-internal adaptation moves to @container.
