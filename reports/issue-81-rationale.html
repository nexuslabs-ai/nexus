Why alpha-variant tokens?

Issue #81 adds a50–a950 to every primitive palette in color.json. The short version: solid colors don't blend. Without alpha tokens, every translucent surface in Nexus is a hand-tuned rgba(...) that looks correct on exactly one background — and wrong on every other.

If we don't do this

SymptomWhat goes wrongWho pays the cost
Hardcoded rgba() in components Designer hand-picks rgba(15, 23, 42, 0.12) for a border. Looks correct on the slate base. Now there are 5 base palettes — wrong on the other 4. Designers re-tune per brand. Engineers fork components. Quality drifts.
Theming impossible Alpha values bake into source code. Switching from slate to stone base means re-grepping every rgba(...). The multi-brand thesis collapses at the translucent layer. Brand work becomes one-offs. Defeats the "AI-native multi-brand" thesis.
No glass / vibrancy effects Apple HIG materials, Linear's Cmd-K, Raycast palettes, Notion mentions, Stripe row hover, Vercel/Geist gray-alpha-N — every Tier-A system uses translucency. Nexus is the only Tier-A-aspiring system without it. Components feel flat next to category benchmarks. Surface credibility erodes.
Designers can't see alpha in Figma Translucent surfaces don't appear in Tokens Studio / Figma Variables. Designers fake them with low-opacity layers or use brand colors that pretend to be alpha. Code-vs-design parity erodes. The Figma audit can't compare what isn't there.
Modal scrims look wrong on tinted pages Today's scrim is #1e1e1ecc — a fixed dark with fixed alpha. On slate it looks correct; on stone or zinc it reads as a foreign object. Per-palette tweaks accumulate in component CSS. Compounding drift.

What changes in code

Before

// component/popover.tsx — today
<div style={{
  background: 'rgba(255, 255, 255, 0.85)',
  border: '1px solid rgba(15, 23, 42, 0.12)',
}}>

// Dark theme? Designer must:
// 1. know slate.900's exact hex
// 2. hand-mix it with 0.85 alpha
// 3. duplicate it in every consumer
// 4. rebuild when base palette swaps

After

// component/popover.tsx — with #81
<div className="nx:bg-popover-alpha
            nx:border-border-default-alpha">

// Light theme: white@90% on surface
// Dark theme: slate-950@85% on surface
// Different base palette? Same token.
// Tokens visible in Figma. Figma audit covers it.

What ships with this PR

  • 264 primitive alpha tokens. 22 chromatic palettes × 11 alpha shades, plus white-a* and black-a* leaves. Mechanical OKLCH conversion preserves alpha — no script changes needed.
  • 50 semantic alpha tokens. 5 alpha keys per base file × 10 base files (5 palettes × light/dark). Covers borders, hover, popover, scrim, backdrop.
  • Generated CSS & utilities. --nx-color-slate-a200, --color-border-default-alpha, etc. Tailwind utilities like nx:bg-popover-alpha become available.
  • Tokens Studio / Figma Variables visibility. Designers can pick alpha tokens directly. Code-vs-Figma audit can compare them once the snapshot is refreshed.
  • Documentation. .claude/rules/tokens.md gets an Alpha Token Scale section explaining the curve and the mechanical routing.
  • Pure addition, zero breakage. No existing token values change; no CSS variables renamed; no Storybook regressions expected. audit:contrast stays green (alpha intentionally out of APCA scope per the issue).

Why now, not later

Three reasons this is the right phase:

  • Theming gate. The 10 base palettes ship in Phase 1. Alpha tokens must exist before components can be themed without hardcoded rgba() — otherwise every later component PR carries the migration cost.
  • OKLCH pipeline is ready. The mechanical routing path already handles 8-digit alpha hex correctly. Adding alpha later would still cost the same — but later means more rgba() drift to clean up.
  • Tier-A parity. Linear, Geist, Radix, Apple HIG, Stripe, Notion all ship alpha. Nexus's "AI-native multi-brand design system for humans and agents" thesis (per memory) requires showing parity with the systems agents will be benchmarked against.
