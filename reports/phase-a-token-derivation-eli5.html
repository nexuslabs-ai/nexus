The Big Idea

ELI5: instead of shipping five different coloring books and hoping the right one loads, we teach the coloring machine how to mix every page itself.

1 Inputs

Brand color, surface tone, light/dark seeds, and contrast.

2 Families

Primary, secondary, status, surfaces, alpha, and charts.

3 Paper

Light mode gets subtle tinted paper so Stone and Slate read.

4 Proof

Key parity, value parity, APCA, colorblind, and contrast gates.

5 Package

Phase B can expose this cleanly to every Nexus consumer.

What We Will Build

The plan is nine small TDD tasks. Each one teaches the engine one missing piece or adds a gate that proves the engine did not drift.

task 1 Reusable family mixer

Turn primary logic into `deriveFamily`, so many color families can share one safe recipe.

task 2 Secondary family

Emit the neutral secondary tokens exactly, instead of treating secondary like a brand ramp.

task 3 Status families

Emit success, warning, error, and information from fixed canonical ramps.

task 4 Surface tone

Add Stone, Neutral, Zinc, Slate, Gray and make light paper visibly tonal.

task 5 Chart colors

Emit a fixed five-color chart set that stays colorblind-safe.

task 6 Alpha tokens

Generate overlays, translucent borders, and popover backdrops with the right ink.

task 7 Key parity gate

Prove the engine emits exactly the same token keys as the curated source.

task 8 Safety sweep

Run APCA, contrast, and colorblind checks across derived tokens.

task 9 Tone parity gate

Freeze light-tone fixture values and prove every tone-owned token passes.

The Light Tone Choice

We keep the baked tone identities. Slate, Gray, and Zinc stay a cool family that differs by intensity; Stone is warm; Neutral is true grey.

Slate h 264.7 lightC 0.011 darkC 0.040
Gray h 261.7 lightC 0.008 darkC 0.027
Zinc h 262.8 lightC 0.005 darkC 0.005
Stone h 70 lightC 0.008 darkC 0.006
Neutral h 0 lightC 0 darkC 0

The Gates

These are the tests that keep the work production-ready. The plan is not done just because colors look good in one screenshot.

Key parity

Derived token names exactly equal curated token names. No missing keys. No extras.

Mode-split values

Dark matches curated. Light matches the new frozen evident-tone fixture.

Contrast

Text and surfaces clear APCA floors. No token wins by lowering the bar.

Colorblind

Chart and status colors stay distinguishable from derived output.

No static leftovers

Anything still relying on base CSS must be explicit, not a silent follow-up.

No browser tricks

`themeToCss` stays plain `:root` and `:root.dark`, with no `light-dark()`.

How We Should Implement It

Do not do this as one giant coding sprint. It is one epic branch, but it should be implemented as four focused sessions with a stop gate after each wave.

Wave 1 Tasks 1-3: family engine, secondary, status. Commit after each task.
Wave 2 Tasks 4-6: surface tone, chart colors, alpha tokens. Review visually after Task 4.
Wave 3 Tasks 7-9: key parity, safety sweep, tone fixture and parity. Expect most debugging here.
Wave 4 Hardening: full checks, PR body refresh, reviewer notes, package handoff readiness.
  • Keep it on one epic branch because tasks depend on the same engine file.
  • Use one commit per task so regressions are easy to bisect.
  • Use read-only council reviews after Wave 2 and Wave 3.
  • Do not start Phase B package APIs until Task 7 and Task 9 are green.
  • Let visual reports guide taste, but let parity tests decide readiness.
