Surface Options — Nexus DS

Three design decisions for surfaces.md / color-shades.md / nav-* tokens. Each mockup uses the actual slate primitives that would ship. Annotations under each option show the token values you'd be committing to.

1 · Nav chrome contrast

How distinct should sidebars and topbars be from the canvas? Each option shows the same SaaS shell — sidebar, topbar, page card, popover — in both modes. Hover/active states use the option's nav tokens.

A · Bold (Linear-style) Recommended

Nav one step darker than canvas in both modes. Clear chrome/content separation; chrome recedes.

2 · Dark-mode container fix

Current dark-mode container = black (#000) is darker than the canvas (slate.900), so "raised" cards visually recess. Research consensus: raised surfaces should be lighter than canvas in dark mode (Figma, Linear, MD3).

Current — container = black Inverted

Card sits darker than canvas. Reads as a hole, not a card.

What I'd recommend after seeing these

  • Nav: A · Bold — most legible chrome/content hierarchy, works for both modes uniformly, matches Linear which is the closest reference for SaaS app shells. C is high-commitment branding; B reads as "no chrome" in light mode and may be confusing once a sidebar lands.
  • Container: Fix now — the inverted elevation is a bug, not a style choice. Cost is one line in each dark-mode base file plus a re-run of the APCA audit.
  • muted-light: Remove — the slot has no role today and adding a real third surface requires either a new primitive or a color-mix() rule we don't yet have consumers for. Pre-production lets us delete cleanly.
