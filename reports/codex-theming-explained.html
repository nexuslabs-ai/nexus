What we built & shipped this session

We taught an app to mix its own colors.

Most apps ship a giant, fixed box of crayons for every theme. We gave Nexus an engine that takes just three colors and a contrast dial, then mixes the other ~100 itself — and repaints the whole UI instantly.

🎨 The buttons in the top-right change this page's colors using the exact idea we shipped — 3 seeds in, everything else derived.

01The big idea

Preset crayons vs. a mixing machine

This is the whole concept Codex (and Linear, Notion, Claude) use — and what we rebuilt for Nexus. Two ways to colour an app:

The old way

A giant box of crayons

Every colour is decided ahead of time — this exact blue for buttons, this exact grey for cards, a hundred values per theme. Switching theme means swapping the entire box. Flexible, but heavy, and you can only pick boxes someone pre-made.

The Codex way — what we built

3 crayons + a machine

You hand over an accent, a background, a text colour and one contrast dial. A small engine (deriveTheme) mixes every other shade — buttons, hovers, borders, faint text — on the spot, and the whole app repaints live.

02How one colour reaches a button

Raw paint → labelled jar → component

Nexus colours travel through three layers. The engine's trick is to repaint the middle layer at runtime — so nothing in any component has to change.

Layer 1 · primitive The raw paint tube --nx-color-blue-600 A fixed swatch on a shelf. Never referenced by components directly.
→
Layer 2 · semantic The labelled jar --nx-color-primary-background A role ("button background") that points at a paint tube.
→
Layer 3 · utility What the button asks for nx:bg-primary-background The component only ever names the role — never a raw colour.
📌

The injection point. The engine writes a tiny stylesheet that re-points the labelled jars (the --nx-color-* layer). Every component pointing at a role updates at once — zero component edits. We made its CSS land on :root / :root.dark so it out-ranks the preset by specificity, not luck.

03The safety referee

It can't derive something you can't read

The machine doesn't just pick pretty shades. Every text-on-background pair is checked by a contrast referee (APCA). If a derived text colour comes out too faint, it's nudged toward the surface until it's legible again — so you literally cannot end up with white-on-white.

Without a referee

Pretty, but unreadable

"Quiet" grey text on a near-grey card looks elegant in a mockup and vanishes on a real screen. A naïve derive engine ships that.

What we built

Quiet, but always legible

Text tiers ramp toward the surface for a soft look, then stop the moment they'd fail the contrast gate. Body text, UI labels and faint metadata each get their own legibility floor.

Why it travels to mobile, too: the engine returns plain data — a list of colour values — not web CSS. A small "applier" turns that data into a stylesheet for the web. The same data could feed a phone app just as easily. That's the cross-platform shape you admired in Linear, Notion and Claude.

05The bugs we squashed

Every fix, explained like you're five

Round 1 — the three big ones
● Blocking

Two bosses, one light switch

Two different pieces of code both flipped "dark mode" on and off. They drifted apart — turn the feature off and the room could stay dark.

Fix →one "arbiter" now owns the switch; the others let go.

● Blocking

A bouncer who only checks your shape

The code that checks saved colours only asked "is this a string?" — so junk like "not-a-color" slipped in, got saved, and crashed the app on reload.

Fix →actually try to read each colour; reject junk; add a safety net.

● Blocking

The preset shouting over the custom theme

With a custom theme on, nudging an old preset dropdown re-injected the preset — and it could win the tie and override your colours.

Fix →lock those dropdowns while custom is on, and turn the derived theme's volume up.

Round 1 — four quick tidy-ups
Minor

Accessibility shouldn't vanish

Font size & reduce-motion were tied to the colour theme, so toggling colours off also dropped them.

Fix →apply those app-wide, always.

Minor

Don't hand out every tool

The engine's public toolbox exposed helpers nobody outside used.

Fix →keep internals internal; expose only what's consumed.

Minor

Follow the system at night

"Match system" read the OS theme once and never listened again.

Fix →subscribe, so an OS flip repaints live.

Minor

Two identical helpers

The same colour-formatting function was copy-pasted in two files.

Fix →extract one shared oklch-format.

Round 2 — cleaning the ripples the fix made
Minor · ripple

A light switch wired to nothing

After Round 1, a helper still had a "remove the style" path that could never run anymore — dead wiring left in the wall.

Fix →remove the dead branch; tighten the type.

Minor

A clamp with the wrong backup number

The font-size safety net fell back to 14 for everything — but code text defaults to 12. Type a "0" and code rendered at 14. And three places clamped three different ways.

Fix →one shared clamp, correct per-field backup, plus a test.

06The one thing to remember
A change isn't done when it compiles. It's done when everything around it is as clean as if you'd written it that way from the start.
