00Executive Summary

The token JSON and OKLCH pipeline are largely sound — primitive ↔ semantic references all resolve, the perceptual L-grid routing matches the documented spec, and theme parity passes across all ten light/dark palette pairs. The cracks appear at the seams: where the build pipeline emits the bundled tailwind package, where components reference tokens that don't exist, and where docs teach patterns the code no longer uses.

Warning

Info
61
Observations, polish, cosmetic, dead code, documentation gaps without immediate impact.

The five highest-impact findings

  1. The bundled @nexus/tailwind shadow utilities are structurally broken. nexus.css emits 42 var(--nx-shadow-*) references; variables.css declares zero. The modular pipeline (playground) works; only the bundled generator drops shadows because it never adapted to themed shadow files. C-01
  2. Badge light-fill variants render unstyled. Seven references to *-text and *-surface tokens that don't exist in nexus.css. The Playground's IconShowcase repeats the same bug pattern. C-04 C-05
  3. @nexus/react ships a broken CSS export. package.json maps ./styles.css → ./dist/style.css; the actual built file is dist/react.css. Any consumer of import '@nexus/react/styles.css' gets a module-resolution error. C-09
  4. CONTRIBUTING.md is structurally obsolete. Teaches imports (axe, render, screen, userEvent from @nexus/test-utils) that don't exist, assertions against class names that have no nx: prefix, and a three-layer testing model that was replaced by story-first play functions. C-12
  5. Focus-ring style diverges three ways across the library. Components mix focus: vs focus-visible:, three different ring colors, and inconsistent opacity tokens. Tabbing through a form changes the indicator color per control. C-13 C-14
How to read this report
Each finding has two parts: "In plain English" — what's wrong, in everyday words, useful for skimming or sharing. "Technical detail" — the file, line, and exact reason, useful when you're fixing the bug. Every finding has a stable ID (e.g. C-01, W-12) so the PR plan in §07 can point back at it. Critical = red, Warning = amber, Info = blue.

1.1Critical — bundled pipeline

Critical
C-01Shadow primitives missing from bundled CSS
packages/tailwind/variables.css ↔ nexus.css

The shipped CSS file uses 42 shadow variables that aren't defined anywhere. Every shadow-sm, shadow-lg, etc. utility silently renders nothing in any app that installs @nexus/tailwind.

nexus.css has 42 var(--nx-shadow-*-layer-N-{x,y,blur,spread,color}) references; variables.css has zero declarations. Confirmed by grep (grep -c '^[[:space:]]*--nx-shadow-' returns 0). Every consumer using nx:shadow-2xs … nx:shadow-2xl gets an unstyled element.

Critical
C-02generate-tailwind-package.js doesn't handle themed primitive files
packages/core/scripts/generate-tailwind-package.js:49,163

The CSS generator was never taught that shadow files now come in light + dark versions. It looks for the old filename, doesn't find it, logs a quiet warning, and ships a broken CSS with no error. This is the root cause of C-01.

generate-modular.js:232 calls partitionThemedModes to handle the shadow-{mode}-{theme}.json shape introduced in PR #34. The bundled generator was never updated. loadTokensWithNxPrefix emits a buried log.warn, the script exits 0, and the broken CSS ships. The shadow regression would be caught by an end-to-end snapshot test, but there is no such test.

Critical
C-03CI doesn't enforce the gates the rules promise
.github/workflows/ci.yml

The rules say CI must block PRs on contrast checks and visual-regression tests. CI never runs those checks. So the "gates" are just suggestions — broken styles can merge undetected.

.claude/rules/tokens.md documents APCA contrast as "enforced"; CI never runs audit:contrast. .claude/rules/testing-react.md documents Chromatic as the visual-regression gate; CI never runs chromatic:ci. CI also doesn't verify that regenerating tokens produces a diff-clean working tree — the shadow break above could have been caught by a regenerate-and-diff job.

1.2Critical — token JSON

Critical
C-15xs shadow spread references the wrong layer
packages/core/tokens/styles/shadows.json:18

An xs-shadow recipe accidentally borrows a value from the 2xs recipe (copy-paste mistake). Today both values happen to be zero, so nobody notices. The moment someone tunes one but not the other, the xs shadow gets the wrong spread silently.

The xs shadow's spread reads {2xs.layer-1.spread} (copy-paste from the 2xs block at line 8). The other 5 properties on the same composite correctly point at {xs.layer-1.*}. Values happen to match today (both are 0px), so the bug is silent — it surfaces the moment any mode diverges.

Critical
C-16tokens.schema.json regex rejects every shipped key
packages/core/tokens.schema.json:7,60

The "validator" that's meant to catch malformed tokens rejects every key that starts with a digit (50, 100, 2xs, etc.) — which is most token keys. So in practice it validates nothing. A safety net with a hole.

The key regex ^\$?[a-zA-Z_][a-zA-Z0-9_-]*$ doesn't allow leading digits (50–950, 0_5, 2xs, 2xl, 3xl). tokens.md documents this file as the validation gate. In practice it validates nothing the codebase actually ships.

1.3Warnings

Warning
W-01APCA gate is missing two documented foreground pairs
packages/core/scripts/audit-contrast.js:20-27

The contrast checker is missing two text-on-background pairs that the rule says to check. Those pairs may be failing accessibility quietly with nobody alerted.

tokens.md:48-52 defines the 45-Lc incidental tier and explicitly includes muted-foreground ↔ muted. BASE_PAIRS covers that one but skips muted-light-foreground ↔ muted-light and disabled-foreground ↔ disabled. The stone-light palette resolves those to stone-400/stone-50 and stone-300/stone-100 — likely below the 45-Lc floor, but never asserted.

Warning
W-02Typography letter-spacing tokens carry Figma float-32 junk
packages/core/tokens/primitives/typography/typography-{vega,lyra}.json + variables.css:341-346

Five letter-spacing values are ugly numbers like -0.800000011920929px instead of -0.8px — leftover precision artifacts from a Figma export. Works correctly, but looks broken when anyone inspects the CSS.

Values like -0.800000011920929px, 0.4000000059604645px, 1.600000023841858px survive the Figma export and propagate verbatim into emitted CSS. The other three modes (maia/mira/nova) use clean values. Rounding belongs in the generator (utils.js), not on a per-token basis.

Warning
W-03Two byte-equivalent reference resolvers
packages/core/scripts/utils.js:171-209

Two functions are byte-for-byte identical except for one comment line. Delete one.

resolveReference and resolveReferenceWithNxPrefix are identical except for a single comment line. Two functions, one implementation. Violates the "fewer moving parts" principle in code-quality.md.

Warning
W-04Multi-layer inset shadows silently drop the inset prefix
packages/core/scripts/utils.js:822-833

If a shadow ever has both multiple layers AND the "inset" property, the generator silently drops "inset". No such shadow exists today; the next one will break silently.

generateShadowVarValue only applies the inset prefix to single-layer shadows. For multi-layer arrays the isInset flag is ignored. Today no shadow uses multi-layer + inset, but the next one will silently fail.

Warning
W-05line-height: auto emitted to CSS
packages/core/scripts/utils.js:604 → typography-utilities.css:135

The generator emits line-height: auto, which browsers don't accept (the CSS spec only allows normal, a number, or a length). Browsers silently ignore it — the setting has no effect.

Source typography.json:195 sets "lineHeight": "auto" for typography-code-inline; the generator passes it through; browsers reject line-height: auto as invalid. Either map auto → normal in resolveTypographyProperty, or fix the source token.

Warning
W-06137 unused color primitives shipped in the bundled CSS
packages/tailwind/variables.css

The CSS file ships 244 color variables but the default theme uses only ~107 of them. The other ~137 (cyan, emerald, fuchsia, etc.) are dead weight that every consumer downloads but never uses.

variables.css emits all 244 color primitives. nexus.css (default stone+neutral theme) references ~107. The remaining ~137 (cyan, emerald, fuchsia, indigo, orange, pink, purple, rose, sky, teal, violet, yellow, zinc, and several partial palettes) are dead in the default bundle. Either tree-shake or accept and document.

Warning
W-07Generation scripts have ~15% test coverage
packages/core/scripts/__tests__/

The scripts that build all the CSS are barely tested. The big shadow bug above (C-01) would have been caught by even a basic snapshot test that compared the new output to a known-good baseline.

audit-contrast.js, generate-tailwind-package.js, generate-modular.js, sync-playground-themes.js have zero tests. Only 9 of 28 utils.js exports are tested. perceptual-grid.js is well-covered (~80%). The shadow regression in C-01 would have been caught by an end-to-end snapshot test.

Warning
W-08utils.js at 1,076 lines / 28 exports has become a library
packages/core/scripts/utils.js

One file ("utils.js") is over a thousand lines and exports 28 different things. It's grown past "utility helpers" into "library that needs structure." Split it into focused modules.

Past the threshold where "utils" is the right name. Should split into tokens.js, css-emitters.js, typography.js, shadows.js, google-fonts.js. Four near-duplicate collect*Tokens functions (spacing/radius/borderwidth) would consolidate to one.

Warning
W-09Stale testing plan with references to deleted scripts
packages/core/scripts/TESTING_ACTION_PLAN.md

A "here's what we'll do" planning doc references script files that were renamed or deleted long ago. Either start executing it or delete it.

"Phase 2/3 pending" since the plan's creation. References generate-css.js and copy-to-react.js — both renamed or deleted. Per project-stage.md, either commit or delete.

2.1Critical findings

Critical
C-04Badge light-fill variants reference 7 non-existent tokens
packages/react/src/components/ui/badge.tsx:36,69,74,85,90,95,100

Seven Badge color styles look up paint colors that aren't in the paint catalog. Those badges render with no background or text color — they're invisible until you put them on a non-white surface.

References primary-text, primary-surface, secondary-text, secondary-surface, error-text, error-surface, warning-*, success-*, information-* in the -text / -surface family. None exist in nexus.css. The actual family is *-subtle / *-subtle-foreground. Visually broken.

Critical
C-13focus vs focus-visible mixing
dialog.tsx:146, select.tsx:72

Two components (Dialog, Select) show a focus glow whenever you click them with the mouse — usually unwanted noise. The other nine components only show it when you tab with the keyboard, which is the modern norm. Inconsistent feel across the same form.

Dialog and Select use nx:focus:ring-*, which lights up on mouse click. Every other component uses nx:focus-visible:, which only shows on keyboard navigation. Users see the indicator inconsistently across the same form.

Critical
C-14Three different focus-ring colors
button.tsx:10, input.tsx:13, switch.tsx:41, dialog.tsx:146, etc.

The focus-ring color is different in three groups of components. Tab through a form and the indicator color shifts as you move between fields. Looks accidental.

ring-primary-background/50 (Button, Accordion, Dialog, Tabs); ring-primary-background-active (Input, Select); ring-primary-background (Switch, no opacity, no -active). Three sources of truth; pick one. See Figure 2.2.

Critical
C-17Input / Select / Tabs use fixed heights
input.tsx:19-22, select.tsx:66, tabs.tsx:52

Three components hard-code their height in pixels (h-10, h-12). The design rule says size with padding instead — fixed heights break in flex/grid layouts and don't scale with font size.

components.md mandates padding-based sizing — fixed heights break in flex layouts. Switch is a legitimate exception (toggle widget). Input/Select/TabsList still ship nx:h-10/h-8/h-12.

Critical
C-18dropdown-menu uses wrong prefix order
dropdown-menu.tsx:92,235

Tailwind classes in one file have the prefix in the wrong position ([&_svg]:nx: instead of nx:[&_svg]:). Tailwind v4 will refuse to compile these classes, so they silently produce no styling.

[&_svg]:nx:pointer-events-none [&_svg]:nx:size-4 — nx: placed after the arbitrary selector. Rule is "prefix always comes first". Tailwind v4 will silently fail to compile these.

Critical
C-19DropdownMenu destructive variant invisible
dropdown-menu.tsx:237-238

The "Delete" option in dropdown menus renders as white text on a white menu background — completely invisible until you hover over it. Should be red text on the default background.

Destructive item uses nx:text-error-foreground (white) with focus:bg-error-background — applies white text without the colored background by default, so the menu item is invisible on a white menu until hovered. Should be text-error-subtle-foreground + focus:bg-error-subtle.

Critical
C-20accordion.tsx uses forwardRef; 12 others use function
packages/react/src/components/ui/accordion.tsx

Twelve components are written as plain functions. The thirteenth (accordion) uses an older React pattern (forwardRef). When a developer copies an existing component as a template, they hit a coin-flip — which is the right pattern?

Confirmed sole user of React.forwardRef + displayName in the library. The template in components.md mandates plain function components. Per ripple-effect.md, change once and bring the rest of the library into one pattern.

Critical
C-21CardAction positioning requires consumer guesswork
card.tsx:162-173, used by Card.stories.tsx:83,324,384

The Card has an "Action" slot positioned to the top-right. But Card itself isn't set up as a positioning container — so every consumer has to remember to add nx:relative to Card themselves. Easy to forget.

CardAction applies nx:absolute nx:right-6 nx:top-6 but Card has no nx:relative. Every consumer must remember to add it themselves; the stories do it three times. Either add nx:relative to Card base, or make CardHeader the positioning context.

Critical
C-22Dialog close-button focus ring uses focus: not focus-visible:
dialog.tsx:146

Dialog's close button flashes a focus ring on mouse click (same bug as C-13). Also missing a label that screen readers can announce.

Same family as C-13; called out here because Dialog also lacks a data-slot on the close icon and the close button itself lacks aria-label (the inner .sr-only span provides the accessible name, but the conventional pattern is on the button).

2.2Warnings

Warning
W-10type = vs interface extends split
8 files use type, 5 use interface

Some component files declare their props with TypeScript's type, others with interface. The rule says interface. Mixed style — pick one.

type: accordion, card, dialog, dropdown-menu, select, switch, tabs, tooltip. interface: alert, avatar, badge, button, input. components.md mandates named interfaces.

Warning
W-11data-variant/data-size coverage is patchy
multiple components

Most components add little tags (data-variant="primary", data-size="lg") so CSS can target specific states. Some components include them, some don't, with no clear rule. Inconsistent surface for theming and testing.

Input has data-size but no variant (no enum variants exist, OK). Alert has variant but no size. Select / Switch / Tooltip have neither. No documented rule for when to omit.

Warning
W-12asChild support is patchy
multiple components

The asChild prop lets a consumer swap the wrapping element — so <Button asChild><a href=...>Link</a></Button> works. Only 2 of 13 components support it; the rest force a fixed element type, limiting composition.

Only Button and Badge implement asChild. Alert, Input, Card sub-components, Dialog content, Tooltip content, Switch don't — yet are likely composition targets. CardTitle specifically hardcodes h3, breaking heading hierarchy in nested layouts.

Warning
W-13destructive vs error variant naming
badge.tsx vs button.tsx, alert.tsx, dropdown-menu.tsx

Most components expose a "delete-style" variant as variant="destructive". Badge calls the same thing variant="error". Inconsistent public API — consumers can't predict the prop value.

shadcn-divergences.md mandates public variant prop name stays destructive for shadcn API compat. Button, Alert, DropdownMenu comply; Badge exposes error. Public API drift.

Warning
W-14badge.tsx duplicates render branches
badge.tsx:182-193, 196-234

Badge's render code has a nested if/else for choosing padding, then duplicates almost the entire return statement for the two element types. Refactor to a single shared body with a swapped tag.

Nested ternary isNumber ? 'p-x' : isCaps ? 'p-y' : 'p-z' for class composition; two near-identical return branches for Slot vs span. Refactor to the Comp = asChild ? Slot : 'span' shared body in the components.md template.

Warning
W-15Long single-string classNames hurt scanability
button.tsx:10, accordion.tsx:75

Some components have 200-character single-line class strings — unreadable. tabs.tsx already uses a cleaner array format. Bring the others into line.

12+ utilities concatenated on one line. tabs.tsx already uses the cleaner array form (['classes', 'more', 'classes']) — bring the others into line.

Warning
W-16Button shows spinner after children
button.tsx:93-101

Loading buttons render "Submitting… ⟳" with the spinner trailing the text. Most patterns lead with the spinner: "⟳ Submitting…". Designer call, but worth flagging.

Loading state renders children then spinner; reads as "Submitting… ⟳". Most patterns lead with the spinner. Flag for designer review.

Warning
W-17DropdownMenuItem hybrid variant pattern
dropdown-menu.tsx:200-205, 237-238

One item's "destructive" style is half-implemented with the standard variant library and half with manual if-checks. Pick one approach end-to-end; the current shape is the worst of both.

Introduces variant: 'default' | 'destructive' as a literal union, not CVA. Then a compound conditional in the class string applies destructive styles. Worst of both worlds: not CVA, not data-attribute styling.

08Glossary

Quick reference for terms used throughout this report.

OKLCH
Perceptually uniform color space (Oklab, with L=lightness, C=chroma, H=hue). Nexus emits colors as oklch(...) at build time; hex stays in JSON because Figma can't round-trip OKLCH on export.
DTCG
W3C Design Tokens Community Group format. Every token has $value + $type; cross-token references use {path.to.token}.
CVA
class-variance-authority. Maps component variants to class strings; used by all enum-style variants in packages/react/src/components/ui/.
nx: prefix
Tailwind v4 variant-style prefix. Every Nexus utility class starts with nx: (e.g., nx:bg-primary-background). The prefix always comes first, before any pseudo-class or arbitrary selector.
Semantic vs primitive tokens
Primitives are raw values (--nx-color-blue-500). Semantics reference primitives by role (--color-primary-background → --nx-color-blue-500). Components consume semantics only.
asChild / Radix Slot
Composition pattern from Radix UI. When asChild is true, the component renders its child instead of its default element, forwarding props and refs. Lets <Button asChild><a>...</a></Button> work.
Radix portal
Many Radix overlay primitives (Dialog, DropdownMenu, Tooltip, Select content) render into document.body via createPortal to escape z-index/clipping problems. This means they're not descendants of the trigger's DOM ancestors — so wrapping a div.dark around stories doesn't reach them.
APCA
Accessible Perceptual Contrast Algorithm. Lc (lightness contrast) score replaces WCAG's contrast ratio for OKLCH-based palettes. Nexus uses thresholds 75 / 60 / 45 by usage tier.
shadcn/ui
Component recipe library Nexus is forked from. Nexus diverges in token names (destructive → error in CSS, retained as destructive in props), prefix (nx: required), and sizing convention (padding-based, no fixed heights).
Story-first testing
Component tests live as play functions inside *.stories.tsx — no separate *.test.tsx files. Imports from storybook/test. Hooks and pure utilities still use Vitest unit tests.
