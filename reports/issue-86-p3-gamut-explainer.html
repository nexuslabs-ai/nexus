Issue #86 · explainer

Deliver P3 wide-gamut OKLCH on capable displays

We're letting Nexus colors use the full range modern screens can show — instead of dimming every vivid hue down to the limits of a 20-year-old monitor. One clamp swap in the build pipeline; everything else stays put.

ELI5 in three lines

  • What — Stop forcing every Nexus shade into the small "sRGB" box of colors. Use the bigger "P3" box that modern phones, laptops, and tablets can actually show.
  • How — Change one line in the color build: clampChroma(…, 'rgb') → 'p3'. Browsers handle the rest automatically — no media queries, no fallbacks.
  • Cost — Tiny and provably safe. Neutrals don't change. Contrast (APCA) is byte-identical. Only some vivid hues — mostly teals, ambers, skies, and chart colors — get richer on P3 displays.

1The big idea

Every screen can only show a fixed range of colors — its gamut. Think of it as a box of crayons:

sRGB — the old 24-crayon box (1996)

P3 — the 48-crayon box (most screens since ~2017)

Every crayon in the 24-box also exists in the 48-box. P3 just adds more saturated versions — extra vividness for greens, teals, reds, oranges. Today the Nexus build was reaching for a P3 crayon, then manually rubbing it on a tissue to dim it to sRGB before handing it over. That stops.

For contrast math (APCA): a separate sRGB clamp

APCA checks if text is readable on a surface. It assumes sRGB. So in the audit path, we add a quiet sRGB clamp just for scoring — so contrast values stay byte-identical to today. Verified: 0 mismatches across all 242 shades. Display gets P3; math gets sRGB; no contrast guarantee is weakened.

5Gaps and issues I found

Color-coded: amber = a real gap found, green = closed/proven, teal = a decision worth flagging, gray = manual step I can't do here.

!

The issue's "throw if any shade is outside P3" is infeasible

I checked every shade against P3 at its pinned lightness. 14 shipped shades are technically outside P3 — including amber.500–950 (warning), red.50/900/950 (error), blue.50/950 (info), and chart colors. These aren't bugs; they're intentional vivid hues that can't hold both their pinned lightness and full chroma inside any gamut. A hard throw would break the build on real palette colors.

Resolution: graceful P3 clamp + a warning when chroma still has to drop >20%. This is also what the original oklch-migration.md plan (line 91) explicitly specified: "warn (not fail)". The issue contradicts the project's own established design intent here.

!

The issue describes code that doesn't exist

The issue says the current clamp is culori.toGamut('rgb')(adjusted). The shipped code uses clampChroma(target, 'oklch', 'rgb'). The distinction matters: clampChroma preserves L exactly; toGamut doesn't. If we'd followed the issue literally, we'd have broken the perceptual-grid pinning.

Resolution: change the gamut argument from 'rgb' to 'p3' on clampChroma — verified empirically that L drift = 0 across all 242 shades.

!

A unit test was about to break silently

perceptual-grid.test.js asserts that #facc15 at shade 400 fires the ">20% chroma drop" warning. Under sRGB it drops 22.0% (fires). Under P3 it drops only 9.9% (won't fire) — the test would have failed.

Resolution: swap the fixture to a real shipped color: amber.950 (#392402), which clips 49.6% under P3 — robust margin, real palette data (testing.md prefers real fixtures over synthetic).

⚙

CI has a freshness guard on three paths, not two

.github/workflows/ci.yml fails if generated CSS is stale across packages/tailwind, apps/playground/public/themes, and apps/playground/src/styles. I'd only listed the first two.

Verified: yarn tokens:tailwind && yarn tokens:modular regenerates all three (the sync-playground-themes.js step copies into both playground dirs). Phase 2 covers it.

!

A third stale doc I'd missed

Beyond tokens.md and color-math.md, docs/plans/oklch-migration.md also describes the sRGB clamp (lines 41, 91). It's a historical plan but is referenced from code.

Resolution: add a minimal forward-pointer ("superseded by #86 — gamut step now targets P3") rather than rewriting the historical decision text. Two living docs stay authoritative.

✓

APCA contrast stability — proven, not assumed

The plan rests on "APCA scores don't change." I verified across all 242 pinned shades that the new path produces sRGB ints byte-identical to today's: 0 mismatches. This directly answers the issue's open question #2.

✓

Lightness preservation — proven, not assumed

clampChroma('p3') never moves L. Verified: max L drift = 0 across all 242 shades. The perceptual-grid pinning the whole system rests on is safe.

6The one decision for you

Throw vs warn on shades that exceed P3? The issue asks for a throw. My data says throw is infeasible (14 shipped shades). My recommendation is to clamp gracefully to P3 and keep the existing ">20% drop" warning, retargeted from sRGB to P3 — which is also what oklch-migration.md originally specified. The PR body will document why the issue's wording was tightened.

Why not just retune the 14 hexes?

Could be done, but it changes what the colors look like — that's a separate scope from "delivery mechanism." It'd reduce chroma on amber/red/blue/orange at deep + edge shades, possibly shift APCA, and require designer review of every retuned primitive. Worth doing if it ever comes up, but not inside a PR whose goal is "deliver P3."
