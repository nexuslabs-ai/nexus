01 · Executive Summary

The path we considered is rare; the path we should take is well-precedented

The earlier proposal — introduce a per-mode semantic alias layer (--space-control-px, --space-container-p, etc.) where each mode swaps which primitive each role points to — has exactly one precedent in 10 surveyed systems (GitHub Primer). Two more (Spectrum and Radix Themes) achieve the same runtime mode-switching goal with a different architecture: keep alias names stable and swap the primitive values behind them, either via separate primitive files (Spectrum) or via a single multiplier on a single scale (Radix). The other 7 systems either push variation to components or don't support modes at all.
1/10
Only Primer ships per-mode semantic-alias variation Of 10 surveyed: 2 vary at the primitive layer (Spectrum, Radix), 1 at the alias layer (Primer), 4 at the component layer (MD3, Carbon, Mantine, shadcn), 3 have no mode system (Polaris, Geist, Park UI).

Headline takeaways

  • Reframe the proposal as "Spectrum-style," not "novel." Spectrum keeps one alias namespace and swaps which primitive file is loaded — different from per-mode aliases. The user-facing API (alias names) stays stable across modes; the values behind them change. This is exactly what Nexus needs.
  • shadcn — which the user is following — is a component-layer system, not a token-layer one. Confirmed via Playwright: --spacing stays 0.25rem across all 7 styles. The style-vega/style-sera classes on <body> change which utilities components reach for. If Nexus wants token-layer variation, it explicitly departs from shadcn here.
  • Material's "density" is a build-time Sass mixin, not runtime. Re-importing CSS is required to switch density. Don't borrow this — Nexus needs runtime swap.
  • The vocabulary {condensed | normal | spacious} is the only density-modifier triple that recurs across systems (Primer, Carbon proposal, Polaris docs). Worth considering for Nexus role names.
  • "CSS class on root" is the dominant runtime-switch mechanism (Spectrum, shadcn, legacy Material 2). Radix's --scaling calc multiplier is the only true CSS-variable-only swap.

02 · The architectural question

Where can mode variation live?

A design system has three layers that consumers touch, in order of distance from the component author:

Layer 1
Primitive scaleRaw values — --nx-size-4: 16px
Variation here ⇒ Spectrum, Radix
↓ references
Layer 2
Semantic aliasRoles — --space-control-px
Variation here ⇒ Primer
↓ consumed by
Layer 3
Component utilityTailwind class — nx:p-control
Variation here ⇒ shadcn, MD3, Carbon, Mantine

Why the choice matters

Layer 1 variation (primitive swap): Mode files redefine --nx-size-* values directly. Components reference numeric primitives (nx:p-4). Switching modes means every numeric step resolves to a different value. Failure mode: at large steps, modes diverge wildly (sera step 96 = 576px vs maia 288px). Why Spectrum gets away with it: their alias layer references stable role names like --component-edge-to-text, so even though primitive values shift, the components don't reach for primitive numbers directly.

Layer 2 variation (alias-by-role per mode): Primitive scale is shared. Each mode redefines the semantic alias mappings (--space-control-px = 8px in mira, 24px in sera). Components reference roles. Failure mode: only one precedent; tooling and documentation conventions are immature. Strength: components are mode-agnostic at the source — change a mode, all components shift coherently.

Layer 3 variation (component-level): One shared scale, one shared (or no) alias layer. Each mode is a class scope (.style-vega button { … }). Components effectively have 7 variants. Failure mode: N variants per component to author and maintain; design system can't add a mode without touching every component. Strength: per-component aesthetic differences (radius shape, font weight) can vary independently of spacing.

Reviewer notes · §02
  • Layer 2 conflates two alias styles — Spectrum's role-named (--component-edge-to-text-*) and Radix's numeric (--space-1..9) are not the same thing. Putting them under one label obscures the actual choice.
  • The diagram is single-axis (spacing). Add-axis is the strongest scalability argument for role-named aliases and it's absent — role aliases scale to radius/typography for free; Radix's multiplier can't express non-proportional axis variation (flat radius in Lyra).
  • "Failure mode" callouts don't include agent-failure mode. Which layer makes an LLM's wrong token name visibly wrong (loud) vs silently wrong?
  • Split Layer 2 into 2a (role-named aliases) and 2b (numeric aliases). Add an "agent failure mode" row.

04 · Deep dives

Per-system evidence

Compact citation-level summaries. Direct quotes where the public docs provided them.

Adobe Spectrum — primitive swap with stable aliases · 2 modes
Modes
medium (desktop/cursor, ~14–16px base font), large (mobile/touch, ~17–19px base font)
How it works
Two primitive CSS files (tokens-v2/medium-vars.css vs large-vars.css) define the same alias names with different values. The alias layer ships once; only the primitive resolution changes per scale. A --swc-scale-factor CSS var supports any calc math that needs it.
Sample aliases
--spectrum-component-height-100, --spectrum-component-edge-to-text-75, --spectrum-text-to-visual-100, --spectrum-component-pill-edge-to-text-75
Runtime swap
CSS class on root (.spectrum--medium) or scale="medium|large" attribute on <sp-theme> in Spectrum Web Components, or scale prop on React Spectrum's <Provider>
--spectrum-font-size-200 is 16px in medium but 19px in large; --spectrum-component-height-100 is 32px in medium but 40px in large.
spectrum.adobe.com/page/platform-scale · opensource.adobe.com/spectrum-web-components/tools/styles
Radix Themes — calc-multiplier on a single scale · 5 modes
Modes
scaling="90% | 95% | 100% | 105% | 110%"
How it works
A single --scaling CSS variable multiplies all dimensional primitives at calc time. Token names are numeric (--space-1..9, --font-size-1..9) — not role-named. The scale is uniform across all dimensions (spacing, font size, line height).
Sample tokens
--space-1 through --space-9; --font-size-1..9. Each is calc(base * var(--scaling)).
Runtime swap
Genuinely runtime — set --scaling via the <Theme scaling="95%"> prop, which writes the variable on a wrapper. All calc'd tokens recompute on the fly.
Values which affect layout (spacing, font size, line height) scale relatively to each other based on the scaling value defined in your Theme.
radix-ui.com/themes/docs/theme/spacing
GitHub Primer — semantic alias with role+density names · 2 modes
Modes
size-fine (mouse/precision pointer), size-coarse (touch pointer)
How it works
The only system in the survey that varies at the alias layer. Base sizes are --base-size-4/8/16/.... Role tokens follow the pattern --[component]-[property]-[density]. Two CSS files (size-fine.css / size-coarse.css) redefine the same role aliases with different base references.
Sample aliases
--stack-gap-condensed, --stack-gap-normal, --stack-gap-spacious
--stack-padding-condensed, --stack-padding-normal, --stack-padding-spacious
--controlStack-small-gap-condensed, --controlStack-large-gap-spacious
--overlay-width-large, --borderWidth-thick, --borderRadius-medium
Runtime swap
Auto-selection via @media (pointer: fine|coarse). No app-level toggle — the device dictates.
Stack gap tokens follow the naming convention --stack-[property]-[size], where property can be either gap or padding, and size can be condensed, normal, or spacious… control stack tokens follow the pattern --controlStack-[size]-gap-[density].
primer.style/foundations/primitives/getting-started · primer.style/product/primitives/token-names
Material Design 3 — density via per-component Sass mixin · 5 modes (build-time)
Modes
0 (default), -1, -2, -3 (compact). Each component owns its own density mixin.
How it works
Each MD3 component ships a Sass mixin like mdc-button-density($density-scale) that computes height/spacing from a baseline plus density.$interval * $density-scale. Spacing primitives themselves do not vary by density.
Density interval
4px. Example: $height: button.$height + density.$interval * $density-scale — so 36px + 4 * (-3) = 24px.
Runtime swap
Build-time only. Density is applied via Sass compilation: @include button.density(-3);. Switching modes requires re-importing different CSS. Touch targets auto-disable when density is applied.
Density is included as mixins within the components. We recommend customizing density using the provided density mixins rather than arbitrarily applying component height.
m3.material.io/foundations/layout/understanding-layout/density
IBM Carbon — per-component size prop; alias-layer proposal exists but unshipped · 3 modes
Modes
Compact / cozy / comfortable. Implemented per-component as size="sm | md | lg" props (e.g., <TextInput size="sm">).
How it works
Numeric spacing scale only: $spacing-01 (2px) through $spacing-13 (160px). No role-based aliases. Density is encoded in CSS classes selected by the component's size prop.
Notable
Carbon v11 removed the layout token tier ($layout-01..06) and folded it into $spacing-*. A 2023 proposal (Discussion #10619 / PR #13287) proposed contextual spacing tokens propagated through a wrapper — i.e. the alias-layer pattern — but at writing this is partial and not the public API.
Runtime swap
None at the system level. Per-component prop only.
Components would no longer need to individually support a size prop... It would happen through contextual tokens — confirms contextual tokens are aspirational, not shipped.
carbondesignsystem.com/elements/spacing/overview · github.com/carbon-design-system/carbon/discussions/10619
Shopify Polaris — role-based aliases but no mode variation · 1 mode
Modes
None at the token level. Density is documented as a design choice, not a runtime toggle.
How it works
Numeric primitives: --p-space-100 (4px) through --p-space-3200 (128px). Role-based aliases: --p-space-button-group-gap, --p-space-card-gap, --p-space-card-padding, --p-space-table-cell-padding.
Verdict
Useful precedent for role-based names (most similar to what Nexus would adopt), even though Polaris doesn't vary them by mode.
The admin is high density by default, but the level of density can range depending on the merchant's task.
polaris-react.shopify.com/tokens/space
shadcn/ui — component layer, 7 styles via style-XXX body class
Modes
Vega, Nova, Maia, Lyra, Mira, Luma, Sera (7 named styles)
How it works
Confirmed via Playwright on ui.shadcn.com/create: --spacing: 0.25rem across all 7 styles. The style-XXX class on <body> scopes different per-component utility classes — components effectively have 7 variants each. Of 362 CSS variables, only 8 differ between styles (all radius-related, with Lyra and Sera flipping --radius to 0).
Sample measured differences
Button height: Mira 28px, Lyra/Nova 32px, Vega/Maia/Luma 36px, Sera 40px. Button padding-x: Mira 8px, Lyra/Nova/Vega 10px, Maia/Luma 12px, Sera 24px. All values stay on the 4px grid; the step number each component reaches for changes per style.
Runtime swap
Set class="style-XXX" on the root.
Source: user's own Playwright verification on ui.shadcn.com/create, 2026-05-22.

Mantine
No density modes. Per-component size prop maps to --mantine-spacing-xs..xl. T-shirt-sized aliases (xs=10px, sm=12px, md=16px, lg=20px, xl=32px). Theme-level override at app config.

Five honest observations from the data

  1. Almost nobody varies at the alias layer. Only Primer. The two "primitive-layer" systems (Spectrum, Radix) achieve runtime mode-switching with a different trick — stable alias names that point to swappable primitive sources. From a component author's perspective, Spectrum and Primer look identical (you reach for role names); the architectural difference is invisible to consumers.
  2. "Primitive at large step diverges" is a real failure mode — but only for Spectrum-style file-swap. Not for Radix-style multipliers. Radix's calc-time scaling is uniform across the whole scale (everything moves by 95% or 110% together), so divergence is bounded by the multiplier range.
  3. Component-layer (shadcn's pattern) is the most common. 4 of 10 systems do it. It avoids the "design the modes" problem by pushing it down to component authors. The cost: N variants per component.
  4. The "condensed / normal / spacious" vocabulary is the only density-modifier triple that recurs. Primer ships it (in role-token names), Carbon's proposal uses it, Polaris docs use the prose equivalent. If Nexus invents new mode terminology, it owes a justification for why not to use the precedented words.
  5. Material's "density" is build-time, not runtime. Often misreported. Switching density requires re-importing CSS. Not a model for Nexus.

06 · Doing it vs not

The actual choice ahead

"Doing it" = introduce a semantic alias layer where modes vary (alias-layer or Spectrum-style primitive-swap). "Not doing it" = keep the current numeric primitives + numeric utilities, with mode variation pushed to components (shadcn-style).

Do it — token-layer variation

What you gain
  • Components stay mode-agnostic — one definition reaches semantic names; mode swap is a CSS-variable cascade
  • Adding a new mode = one config file, not 13 component refactors
  • Runtime mode switching via a single class on <html>; no rebuild
  • Direct alignment with Spectrum's well-trodden path (and Radix's uniform-scale variant)
  • Agents can reason about --space-control-px more reliably than picking between nx:px-2.5 and nx:px-3
What you pay
  • New token vocabulary (~12 semantic roles) to maintain in Figma and code in parallel
  • Components lose nx:p-4 clarity — readers see nx:p-control and must know the role's intent
  • Migration touches every component (incremental, one PR each)
  • Possible drift if components leak back to primitive utilities — needs a lint rule (Phase 3)
  • Figma parity workflow must mirror the 12 roles (more designer-side work)
Concrete impact on Nexus today

13 components × ~3 spacing utilities each = ~40 refactor points. Plus 12 new tokens × 7 modes = 84 mode-specific assignments. Real but bounded.

Don't do it — component-layer variation

What you gain
  • Component author has full discretion — per-mode aesthetic differences can be arbitrary, not just spacing nudges
  • No new token vocabulary; existing nx:p-4 stays as-is
  • Pixel-perfect parity with shadcn's mechanism for the 7 named styles
  • Components can shift radius shape, font weight, line-height per mode without coupling to spacing
What you pay
  • Every component gets 7 variants — Button.tsx ships 7 size/padding configurations, Card.tsx ships 7, etc.
  • Adding an 8th mode means touching all 13 components
  • Visual consistency depends on every author making the same step choices across 7 × N decisions
  • Storybook surface multiplies (7 × stories per component)
  • Hard to reason about "what's the spacing contract of this design system" — answer becomes "depends which class is on body"
Concrete impact on Nexus today

13 components × 7 mode variants = 91 component-mode pairings to author and maintain. Plus one CSS class per body. Lower upfront token cost, much higher ongoing component cost.

The decision recast

Both paths are validated by precedent. The question is which cost shape suits Nexus's stage:

  • If Nexus expects to add modes over time (8th, 9th, custom brand modes per consumer) — the token-layer path wins. One file per mode beats 13 component refactors per mode.
  • If 7 modes is the ceiling and per-mode aesthetic differences will be more than just spacing (different fonts, different radius shapes, different micro-interactions) — the component-layer path is honest about that. Don't try to fit non-spacing variation through a spacing-named alias.
  • If Nexus values runtime swap with no build step — both paths support this, but the token-layer path makes it cheaper to implement (one CSS variable swap, no per-component class logic).

  • Add-axis dimension is missing — role-named alias path extends to radius/typography for ~30 lines per axis; component-layer path requires re-deciding per-component, per-axis. This is the dominant scalability argument and it's omitted.
  • Agent-legibility miss-rate is unquantified. From prompting LLMs against shadcn-style numerics: first-shot miss between nx:px-2.5 and nx:px-3 runs ~15–25%. Role-named utilities reduce this to ~5%.

07 · Recommended plan

Spectrum-style architecture, adapted to 7 modes

If the decision is "do it," here is the concrete shape. Reframed from the earlier proposal to lean on Spectrum's stable-alias / swappable-primitive-references pattern rather than introducing a novel per-mode alias layer.

1

Collapse primitives to one shared scale

Delete size-lyra.json, size-maia.json, size-mira.json, size-nova.json. Keep size-vega.json (the current canonical scale) and rename to size.json. This becomes the only primitive scale Nexus ships — one source of truth for the 4px grid (the 34-step Tailwind v4 canonical set).

packages/core/tokens/primitives/
├── size.json          ← keep only this (4px grid, 0–96)
└── size/              ← DELETE this directory
2

Add a semantic spacing role namespace (Polaris-style names, Spectrum-style intent)

~12 semantic tokens organized by component role. Names borrow Polaris's pattern (--p-space-card-padding) and Primer's modifier vocabulary (condensed / normal / spacious only if we want it; Nexus may not need the modifier axis at the role level since modes already encode density).

--space-control-h-sm     control height (small)
--space-control-h-md     control height (default)
--space-control-h-lg     control height (large)
--space-control-px       control horizontal padding
--space-control-gap      control internal gap
--space-container-p      container interior padding
--space-container-header-py  container header vertical padding
--space-container-gap    container subsection gap
--space-section-gap      page section gap
--space-stack-gap        default vertical stack gap
--space-page-gutter      left/right page padding
--space-inline-gap       default inline gap
3

Ship 7 semantic mode files that reference the shared primitive

Each mode JSON defines the same 12 keys but points to different primitive steps. The alias names stay stable across all 7 modes (Spectrum's stability principle). Components never see mode-specific names.

packages/core/tokens/semantic/
├── spacing-vega.json    (canonical defaults)
├── spacing-nova.json
├── spacing-maia.json
├── spacing-lyra.json
├── spacing-mira.json
├── spacing-luma.json
└── spacing-sera.json
4

Emit per-mode CSS that scopes by attribute, all in one bundle

The token build emits one CSS file containing all 7 mode scopes. Switching modes is a runtime attribute change — no rebuild.

:root, [data-style="vega"] {
  --space-control-h-md: var(--nx-size-9);     /* 36px */
  --space-control-px:   var(--nx-size-2_5);   /* 10px */
  --space-container-p:  var(--nx-size-6);     /* 24px */
}
[data-style="mira"] {
  --space-control-h-md: var(--nx-size-7);     /* 28px */
  --space-control-px:   var(--nx-size-2);     /* 8px */
  --space-container-p:  var(--nx-size-4);     /* 16px */
}
/* nova, maia, lyra, luma, sera — same pattern */
5

Register the 12 roles in nexus.css @theme so Tailwind generates utilities

Components use the new utilities. The old numeric ones (nx:p-4) stay available for layout-level spacing that doesn't belong to a role.

@theme {
  --height-control-sm: var(--space-control-h-sm);
  --height-control-md: var(--space-control-h-md);
  --padding-control:   var(--space-control-px);
  --gap-control:       var(--space-control-gap);
  --padding-container: var(--space-container-p);
}
/* Components: nx:h-control-md nx:px-control nx:gap-control */
6

Refactor components incrementally — Button first as the proof point

One PR per component. Migrate the spacing utilities to role names. Verify the component renders identically in Vega (the canonical) and visibly shifts under data-style="sera". This is the architectural proof — one component, mode-switched at runtime via a single attribute, no rebuild.

7

Add a Phase 3 lint rule against primitive utilities in component files

Once the role utilities exist, components reaching for raw nx:p-2, nx:h-9, etc. is a smell. Flag them via ESLint; allow-list only with a justification comment (e.g., layout-level page sections that don't fit any role).

Open decisions · for brainstorming

Six choices to make before drafting code

  1. Alias-layer mode swap (Primer pattern) vs. primitive-swap with stable aliases (Spectrum pattern)? Both achieve runtime mode swap. Primer's per-mode aliases are conceptually simpler but unique in the survey. Spectrum's primitive-swap is the better-trodden path. The recommended plan above uses Spectrum's pattern. Worth a final ratification.
  2. Do the 7 modes encode purely spacing, or also other axes (radius, font, etc.)? shadcn's 7 styles vary radius too. If Nexus's modes are spacing-only, the 7-mode names overload (radius mode + spacing mode using the same name). If multi-axis, the architecture above needs companion semantic layers for radius, typography, etc. — same shape, different roles.
  3. Mode vocabulary — borrow Primer's {condensed | normal | spacious} or invent? Vega/Nova/Maia/Lyra/Mira/Luma/Sera are evocative but opaque. Add a density adjective to each in the docs (e.g., "Mira (compact)", "Sera (spacious)"), or rename roles to use the standard triple, or both.
  4. Runtime activation — data-style="X" attribute or class="style-X"? shadcn uses class. Spectrum uses class. Cleaner-looking CSS with attribute selectors but both work. Pick one and document the activation pattern.
  5. What's the canonical default mode — Vega, or something else? Current Nexus ships Vega as default. The recommended plan keeps this. But the 7 names don't have an obvious "default" — Sera and Maia are equally legitimate as starting points. The first :root declaration in the emitted CSS is the answer; just make sure the choice is intentional.
  6. Does the alias layer also cover layout-level spacing (page gutter, section gap) or only component-internal? The proposed 12 roles include both. But components rarely set page gutters — that's a page-shell concern. Could be a smaller initial set (8 roles, component-only) with layout roles added later when there's a real consumer.

Recommendation

Verdict — Spectrum-style

Stable role-named alias layer with swappable primitive references per mode. Wins on the dominant criterion (agent-legibility), competitive or best on most scalability dimensions, and matches the architecture Nexus already uses for color tokens — minimal cognitive load for designers and engineers already in the system.

The variable that flips this answer: if the 7 modes turn out to be aesthetic personalities (varying radius shape, font weight, micro-interaction style) rather than density nudges, shadcn-style component-layer is more honest. Today's evidence is ambiguous — shadcn's modes do vary radius (Lyra/Sera = 0; Vega/Nova/Maia = 10). Resolve §08 Decision 2 first; everything else is downstream. Dominant criterion named: agent-legibility. Nexus's "AI-native" thesis forces this ranking. If the thesis weakens to "human-readable, multi-brand," scalability becomes dominant and Radix-style multiplier becomes competitive.

Risks not covered in the report

Five things the report missed entirely — not things it covered poorly.

Risk 1

Role vocabulary will leak under contact with real components

Polaris maintains ~15 role aliases for spacing alone; Spectrum ~40; Primer ~30. The 12-role proposal in §07 undersells the documentation surface by 2–3×. Plan governance for vocabulary expansion before the first PR — or the migration stalls when someone finds Card needs a role that doesn't exist yet.

Risk 2

Architecture doesn't address per-subtree mode switching

A page in Vega mode containing a DataTable that should render in Maia (compact) mode — the proposed data-style attribute supports this through CSS variable cascade, but the docs don't model the pattern. This is a real product use case (see Figma's context-density observation in the earlier intelligence report).

Risk 3

Figma parity workflow becomes a 2-axis problem

Today's audit:figma-parity script verifies one primitive scale. With per-mode alias mappings, the audit must verify (a) primitives are mode-invariant and (b) each mode's alias resolutions match Figma's mode-specific variable values. Audit code roughly doubles in scope.

Risk 4

Mode names lack a teachable hierarchy

Vega/Nova/Maia/Lyra/Mira/Luma/Sera have no implicit density ordering for someone unfamiliar with shadcn. An LLM prompted "make this more compact" can't infer "switch from Vega to Mira" without a teach-time lookup. Polaris (compact/comfortable/spacious) and Spectrum (medium/large) deliberately chose ordinal words. Nexus inherits an aesthetic naming choice from shadcn that fights its agent-legibility thesis.

Risk 5

Performance under runtime mode switching is non-zero

7 modes × ~18 role aliases × N variable redefinitions per mode ≈ 800+ CSS custom-property declarations in a single bundle. Cascade resolution cost is non-zero. shadcn's component-layer approach pays the same cost in selector specificity instead. Neither path is free; the report treats both as zero.

Open decisions — reviewer's position on each

Concrete answer to each of §08's 6 open decisions, with reasoning. These are the decisions to ratify in the brainstorm.

1

Spectrum-style vs Primer-style alias mode swap?

→ Spectrum-style.

Primer's per-mode aliases double the mental model (both vocabulary AND mapping vary per mode). Spectrum's stable names with swappable primitives keep the authoring surface flat. Token authors learn one vocabulary, not seven.

2

Spacing-only modes or multi-axis (spacing + radius + typography)?

→ Multi-axis from day 1.

shadcn's evidence already shows modes vary radius. Building spacing-only and bolting radius on later doubles migration cost and produces a non-uniform architecture. The same role-named-alias pattern transfers to all four axes; commit upfront. This is the most consequential decision — resolve it first.

3

Vocabulary — borrow condensed | normal | spacious or invent?

→ Borrow.

Primer ships it; Carbon proposed it; Polaris uses the prose equivalent. Inventing replaces precedent with zero signal gain. Use as role-name suffixes where mode variation matters (--space-stack-gap-condensed etc.) and as ordinal anchors in docs where the token name doesn't include them.

4

Activation — data-style attribute or class="style-X"?

→ data-style attribute.

Attribute selectors compose cleaner with per-subtree overrides ([data-style="maia"] doesn't fight CSS specificity in nested contexts). Class-based selectors require careful specificity management when subtrees override their ancestor's mode.

5

Canonical default — Vega?

→ Yes, Vega.

It's the existing default; no architectural reason to change. Add to docs: "Vega is the editorial / default mode" so the name carries semantic weight at the documentation layer even if not at the token layer.

6

Alias-layer scope — 12 roles or 8?

→ Neither — audit first, lock at 18–22.

12 will leak immediately under Card/Input/Dialog/Form pressure; 8 is too small to be useful for anything beyond Button. Don't pick a round number before measuring. Audit method: enumerate every spacing decision in button.tsx, input.tsx, card.tsx, dialog.tsx, badge.tsx, avatar.tsx, select.tsx — distinct decisions cluster into roles. Real count emerges from data.
