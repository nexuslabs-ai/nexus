02 · The shift

Drop the primitive layer. Direct values per mode.

Today, primitives are mode-variant (5 size files, all numeric utilities shift implicitly). Proposed: delete the primitive layer entirely. Each mode is one self-contained file with direct px values for both numeric tokens (--spacing-N) and role-named tokens (--control-padding-x-md). No cross-file indirection. No primitive scale to maintain. No "which step does this role point to" gate.

Canonical step set documented in CLAUDE.md, enforced by lint.
Why no primitives
A primitive layer enforces a shared scale but creates cross-mode coupling — editing --nx-size-9 for one mode breaks every other mode that references it. Workarounds (additive primitives, per-mode remapping) add cognitive load without earning their place. Each mode being self-contained with direct values eliminates the coupling and keeps the mental model flat. The scale discipline moves from "the primitive list exists" to "the canonical step set is documented and lint-enforced."

04 · Apples-to-apples · Current vs Proposed

"Role names are more typing." Yes — 4 keystrokes more. Here's what those 4 keystrokes buy you.

The honest count of what changes with role-named utilities: ~4 characters added per utility. ~3 decisions eliminated per component. ~12 things-to-remember dropped from your head. The new vocabulary is smaller than the old one, not larger.

The keystroke trade

nx:px-2.5
9 characters · today
→
nx:px-control
13 characters · proposed
What those 4 keystrokes get you
  • Zero step-picking decisions ("is it 2.5 or 3?")
  • Self-documenting at code review (intent visible)
  • Mode-resilient (same code works in 7 modes)
  • Discoverable via autocomplete (role names cluster)
  • Refactor-safe (rename role, all uses update)
  • LLM-resolvable (intent → role, no guessing)

Vocabulary you have to remember

Today (numeric)

~34 steps × N coupling rules
46+
  • 34 numeric steps in the scale (0, 0.5, 1, 1.5, … 96)
  • Which step is Button padding? px-2.5? px-3?
  • Which step is Input padding? Same as Button or different?
  • Which step is Card padding? p-6? p-8?
  • Whether your numbers match Figma spec exactly
  • Whether other Buttons in the codebase use the same step
  • Whether half-steps are allowed (p-2.5) or forbidden
  • What changes when mode swaps (nothing — it breaks)
  • Which utilities are layout-scale vs component-scale
  • Coupling rules across 13 components

Tomorrow (role-named)

~18–22 named roles
20
  • control-* roles for buttons / inputs / selects / chips
  • container-* roles for cards / panels / surfaces
  • section-* / stack-* roles for layout
  • Which role this component plays — that's it
  • Numeric scale only for layout escape hatches
The counter-intuitive finding
Role-named tokens look like more vocabulary because the names are longer. They're actually less: the new system has ~20 names you can keep in your head; the old system has 34 numeric steps + the implicit coupling rules ("Button uses 2.5, Input uses 3, why?") that nobody documents and everyone has to discover by reading other components.

The 6-months-later code-review test

Someone (or you, in 6 months) opens this file. Read the className. What is this component doing?

Today · numeric 
<button className="nx:h-9 nx:px-2.5 nx:gap-2 nx:text-sm nx:rounded-md">
  Save
</button>
What is this component?
A 36px-tall element with 10px horizontal padding, 8px gap, 14px text, and 6px radius. You can't tell from reading this whether it's a button, an input, a chip, or a tag. You have to check the JSX tag (<button>) — and even then you don't know if these step numbers are intentional for a button or copy-pasted from somewhere else.
mental cost: [lookup h-9 = ?px] [lookup px-2.5 = ?px] [lookup gap-2 = ?px] [verify these are the canonical Button steps]
Tomorrow · role-named 
<button className="nx:h-control-md nx:px-control nx:gap-control nx:text-control nx:rounded-control">
  Save
</button>
What is this component?
A control surface at default size. Self-evident at a glance. The actual pixel values live in the spacing mode file; the component declares what it is, not which numbers it picked. Reviewing this in a PR takes seconds; you check whether the role assignments match the component's actual purpose, not whether the numbers are right.
mental cost: [parse "control surface, default size"] — done.

Real maintenance scenario

A common request from design: "Make all buttons 2px taller in the default mode." Here's what happens.

Task
Designer: "Bump default button height from 36px to 38px"
Today (numeric utilities)
  1. Grep for nx:h-9 across the codebase
  2. For each match, decide: is this a Button? An Input? A Select? Something else?
  3. Filter to only the Button-like usages (manual judgement)
  4. Change them to nx:h-[38px] (no step exists for 38px) or add a new --nx-size-9_5 primitive
  5. Test in all 7 modes — except numeric utilities don't shift with mode, so half the work is mode-by-mode
  6. Hope you didn't miss any Button-like usages
~2–4 hours · drift risk · likely follow-up PR
Tomorrow (role-named)
  1. Open spacing-vega.json
  2. Change one line: control-h-md: size.9 → size.9_5 (or new step)
  3. Every control surface in every component updates
~10 minutes · zero drift · single source of truth
The receipt
Today's system feels familiar because you've internalized the lookups. You memorize which numeric steps belong to which components, and the burden is invisible to you because it's already paid. To a new author, a designer, or an LLM, the lookups are pure tax. Role names eliminate the tax — same scale, fewer decisions, better self-documentation. The "more typing" cost is 4 characters; the "more thinking" cost goes down, not up.
05 · The agent-legibility test

What does an LLM generate for "make this button 36px tall in compact mode"?

Nexus's thesis is human + agent. Every architecture choice gets graded on what an LLM produces with imperfect docs. Failures should be loud (devtools shows undefined CSS) not silent (renders something plausible-looking but wrong).

"Render a 36px-tall Button labelled 'Save' in compact mode."
Spectrum-style (proposed)
<Button className="nx:h-control-md"
  data-style="mira" />
✓ Correct. Role name resolves to 28px under data-style="mira". Switching modes does the right thing.
Primer-style
<Button
  className="nx:h-control-condensed" />
✓ Correct. Density modifier in the name itself. Most self-documenting.
shadcn-style (today)
<Button className="nx:h-9" />
✗ Wrong. 36px is the right answer in Vega but ignores "compact mode." LLM doesn't know to scope inside style-mira class. Silent failure.
Radix-style
<Theme scaling="80%">
  <Button className="nx:h-9" />
</Theme>
✗ Wrong. LLM guesses a multiplier; 0.8 × 36 = 28.8px ≠ 28px. Silent rendering off-grid.
06 · The four architectural options

Where mode variation lives.

Surveyed 10 design systems. Three viable patterns plus shadcn's component-layer (what we copy today). Spectrum-style wins on the criterion that matters most for an AI-native system.

Recommended
Spectrum-style
Stable alias names · swappable primitive refs
Wins
  • Role names are agent-legible
  • Invented names fail loudly (undefined var)
  • Adds new mode in ~50 lines of JSON
  • Matches the pattern Nexus color tokens already use
Costs
  • New role vocabulary to maintain (~18–22 roles)
  • Migration touches every component
Alternative
Primer-style
Per-mode alias maps · density in names
Wins
  • Most self-documenting names
  • Density modifier baked into token
Costs
  • Vocabulary doubles (roles × modifiers)
  • Only one precedent (Primer itself)
Status quo
shadcn-style
Component layer · style-XXX body class
Wins
  • Most flexible per-mode aesthetics
  • Matches what we're doing today
Costs
  • 7 variants per component × 13 components = 91 pairings
  • Silent LLM failures (numeric guess resolves but is wrong)
  • Adding 8th mode = refactor everything
Mismatched
Radix-style
Single calc multiplier · uniform scale
Wins
  • One variable explains everything
  • Trivial runtime override
Costs
  • Can't express non-proportional modes (flat radius in Lyra)
  • Numeric names — same LLM-legibility problem as today

08 · Migration plan

Four weeks. Eleven steps. Button first.

Step 1 (delete primitives) is unblocking. Then audit, document the canonical scale, author mode files with direct values, refactor one component as proof, roll out, then add the discipline tooling that replaces what primitives used to enforce.

1
Week 1 · Foundations
Delete the primitive size layer entirely
Remove packages/core/tokens/primitives/size/ (all 5 mode files). Remove --nx-size-* emission from the build. The scale moves to documentation + lint, not a CSS variable layer.
~1 hour · 1 PR
2
Week 1 · Foundations
Audit existing components for spacing roles
Enumerate every spacing decision in button, input, card, dialog, badge, avatar, select. Cluster into roles. Real count: 18–22, not the 12 the deep-dive originally proposed.
~2 hours
3
Week 1 · Discipline
Document the canonical step set in CLAUDE.md
Without primitives, the "allowed values" rule must live as a project rule. Publish the canonical step set (0, 2, 4, 6, 8, 10, 12, 14, 16, 20, 24, 28, 32, 36, 40, 48, 56, 64, 80, 96, 128, …) as the scale that lint, code review, and LLM context all reference.
~1 hour
4
Week 2 · Tokens
Author 7 mode files with direct values
Each spacing-{mode}.json contains every numeric step (spacing-0 through spacing-96) AND every role (control-*, container-*, …) with direct px values. Self-contained per mode. Each ~80 lines.
~4 hours
5
Week 2 · Tokens
Register tokens in nexus.css @theme
Wire both numeric tokens (--spacing-N) and role tokens so utilities like nx:p-2 AND nx:px-control are legal. Emit per-mode [data-style="X"] scopes in one bundle. No primitive layer to pass through.
~2 hours
6
Week 3 · Proof point
Refactor Button to use role names
One component, one PR. Verify identical render in Vega; add a Storybook story that switches to Sera and visibly shifts. This is the architecture demo.
~4 hours · 1 PR
7
Week 3 · Rollout
Roll out remaining components
Order: Input → Card → Dialog → Select → rest. Each one PR. Components mix role names (for control internals) with numeric utilities (for layout) freely. Both shift with mode.
~1–2 days
8
Week 4 · Documentation
Publish role-to-component coupling table
Single source of truth in CLAUDE.md and figma.md. Documents which role each component uses for each spacing decision. LLMs and humans share this lookup.
~2 hours
9
Week 4 · Guardrails
Lint rule for canonical step set
ESLint/Stylelint flags spacing values in mode files that fall outside the canonical step set, AND flags raw numeric utilities (nx:p-N) inside component files when a role would apply. Prevents drift both ways.
~3 hours
10
Week 4 · Tooling
Refactor audit:figma-parity for mode-level comparison
Existing script audits primitives. Without primitives, the audit shifts to comparing each mode's semantic values against the corresponding Figma collection mode. Same intent, mode-scoped diff.
~3 hours
11
Week 4 · Schema
JSON schema validation for mode files
All 7 spacing-{mode}.json files must have identical key sets. Catches forgotten keys, typos, and key drift across modes at build time. Ensures every mode covers every numeric step and every role.
~2 hours
09 · Decisions

Five open · two now resolved.

Two decisions have landed since the original draft: no primitive size layer (each mode is self-contained with direct values) and both numeric + role-named utilities coexist (component authors pick by intent). Remaining decisions still need brainstorm. Recommended position + flip condition listed for each.

Resolved this round
✓ No primitive size layer. Mode files own values directly. Discipline moves to lint + CLAUDE.md canonical step set.
✓ Numeric + role utilities coexist. nx:p-2 for layout-level spacing; nx:px-control for component-internal contracts. Both shift per mode.
Decision 1 · Dominant
Multi-axis modes (spacing + radius + typography + shadow) — or spacing-only?
Recommended
→ Multi-axis from day 1
Flips if
…the 7 modes turn out to be pure density nudges and we never want radius/font to vary per mode. shadcn's evidence says we already do.
Decision 2
How to keep mode files' values from drifting away from the canonical step set?
Recommended
→ Lint + JSON schema + CLAUDE.md
Flips if
…the team explicitly wants more freedom to use arbitrary values per mode. Then drop the canonical step set; accept that scale becomes implicit.
Decision 3
Density vocabulary — borrow condensed / normal / spacious or invent?
Recommended
→ Borrow
Flips if
…we'd rather lean entirely on the star names (Mira = compact, Sera = spacious) for vocabulary. Then no English modifier ever appears in a token name.
Decision 4
Runtime activation — data-style attribute or class="style-X"?
Recommended
→ data-style
Flips if
…we'd rather match shadcn's exact mechanism for migration ergonomics. Class wins on familiarity; attribute wins on per-subtree composition.
Decision 5
Canonical default — Vega?
Recommended
→ Yes, Vega
Flips if
…most product surfaces ship in Maia or Nova in practice. Default should match dominant use, not first listing.
Decision 6
Role-layer scope — 12 roles or 18–22?
Recommended
→ Audit first; expect 18–22
Flips if
…the audit returns fewer distinct roles than expected. Set the floor by data, not estimation.
